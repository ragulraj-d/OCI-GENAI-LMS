Part 3

Why Chain Is Not Enough

Most important question of the day. We have chains from Day 35 — why do we need something more?

Chain — The Manual Follower

Ramesh writes a step-by-step manual: Step 1 check HR. Step 2 check Finance. Step 3 check OCI. Employee follows exactly. Every time. No thinking.

CFO asks: "Just cloud costs. Skip the rest."

New Joiner: "Manual says HR first. I must follow the manual." Cannot adapt. You must rewrite the manual. That is Chain.

Agent — The Experienced Thinker

Ramesh says one sentence: "Prepare cost report for October." Nothing else.

Experienced person decides: check HR → check Finance → check OCI → calculate → write report.

CFO asks different question? "Just cloud costs." → "Sure, only check OCI." Adapts instantly.

No new instructions needed. That is an Agent.

✅

In one line: Chain = You define the steps.   Agent = AI defines the steps.

Part 4

What Is an Agent — Three Levels

For Common Man

GPS analogy. Old map: you decide every turn. GPS: you say "Take me to Chennai." GPS plans route, reroutes if blocked. You only said the destination. Agent = GPS for business tasks.

For Business Person

Smart virtual employee. Give it a goal. It checks right systems, collects right data, writes the report. You review and approve. AI handles steps. You handle decisions.

For Technical Person

Agent = LLM + Tools + Loop. LLM decides what to do next. Tools are Python functions that do real work. Loop: Thought → Action → Observe. Repeats until done or max_iterations reached.

⚠️

Critical distinction: LLM never executes Python code. LLM only decides which code to run. Python actually executes it. LLM reads the string result. Understanding this separation fixes 80% of debugging confusion.

Part 7

Agent Lifecycle — Configuration vs Runtime

🔑

Key distinction: initialize_agent() is configuration — happens once at setup. agent.invoke() is runtime — happens every time you call the agent. Like hiring an employee (once) vs them doing work (every day).

Configuration — Happens Once at Setup

Analogy — Hiring a New Employee

When you hire someone you give them: appointment letter (system prompt — who they are and what rules), access card (tools — what systems they can use), notepad (memory — what they remember), training (LLM — how they think).

This happens ONCE when they join. Not every morning when they arrive. initialize_agent() = the joining process. agent.invoke() = them actually working.

Runtime Stage 1 — Planning

Analogy — Thinking Before Cooking

Before touching the stove: "I need onions, chicken, rice, spices. I'll fry onions first, then chicken, then rice." Not cooking yet — just planning the sequence in your head.

Agent reads goal + all tool descriptions. Creates a mental sequence. Happens inside Cohere LLM on OCI. You don't write the plan. AI creates it from your goal.

Runtime Stage 2 — Tool Invocation

Analogy — Using PhonePe to Pay Bill

You need to pay electricity. You don't magically pay by thinking about it. You open PhonePe (the tool), select bill payment, enter amount. PhonePe does the actual work. You invoked the tool.

Agent does the same: needs HR data → opens CheckHR tool → passes "all" → gets "35 employees" back. Python runs the function. LLM reads the string result.

Runtime Stage 3 — Reasoning

Analogy — Tasting While Cooking

After frying onions you taste: "Needs more oil." Add oil. After adding chicken: "Needs salt." You evaluate after every step, not just at the start.

Agent after every tool result: "Have HR. Need Finance still." → "Have Finance. Need Cloud still." → "Have everything. Ready to write final answer." This is the Reasoning in ReAct.

Runtime Stage 4 — Output

Agent has all tool results collected. Cohere LLM on OCI writes the final structured answer. LangChain returns it to you.

🛑

Always set max_iterations=7 or 8. Without it, agent can loop forever calling tools without stopping. With it, agent stops after N steps and returns what it has. This is not optional. Always set it. No exceptions.

Part 9

Planning vs Reasoning — The Difference

Planning

When: Once. Before any tool is called.

What: "I will check HR, then Finance, then Cloud, then calculate total."

Where: Inside Cohere LLM on OCI.

Analogy: Deciding the full recipe before touching the stove. Not cooking yet — just planning the sequence ahead of time.

Reasoning

When: After EVERY tool result. Multiple times.

What: "Have HR now. Still need Finance." / "Have everything. Ready to answer."

Where: Inside Cohere LLM on OCI.

Analogy: Tasting the curry after every step. "Needs salt." Add salt. Taste again. "Good now." Continuous evaluation.

Part 10

Tool Selection — How AI Picks the Right Tool

ℹ️

Tool description is the most important signal — but not the only one. LLM also considers: tool name, user query, system prompt, previous observations, and conversation context. Write excellent descriptions and the rest follows.

❌ Poor Description

Tool(name="GetData",
  description="gets data",
  func=check_hr)

Too vague. LLM confused. May pick this tool for any question. Wrong results. Hard to debug.

✅ Good Description

Tool(name="CheckHR",
  description="""Get employee
  count and team size.
  Input: project name or 'all'.
  Use for headcount questions.
  NOT for costs or cloud data.""",
  func=check_hr)

When goal says "How many people in ProjectA?" — LLM reads all descriptions. CheckHR says "employee count and team size" → MATCH → picked. CheckLicenses says "software costs" → no match → not picked. Correct every time.

Part 11

Types of Agents

Task Agent

One specific job repeated many times. Like a specialist doctor who only treats one condition. Example: Resume screener — JD + resume in, fit score out. Use when: one well-defined repeated task.

Tool-Calling Agent

Many tools, AI picks right one per question. Like a general physician — you come with any problem, doctor decides what examination to run. Example: Ramesh's IT cost agent. Use when: questions vary.

Orchestration Agent

Coordinates OTHER agents. Like a hospital CEO who coordinates cardiologist, neurologist, surgeon. Does not do work itself — delegates and combines results. Use when: complex workflow with many specialists.

Multi-Step Planner

Plans ALL steps first, then executes. Like an architect drawing the full blueprint before construction starts. You see the complete plan before anything runs. Use when: long tasks need full visibility upfront.

Part 13

Memory — Why First, Then Types

Why Does an Agent Need Memory?

❌ Without Memory

Turn 1: "I manage ProjectA with 12 people."

Turn 2: "What is cost per person at 120000 budget?"

Agent: "Which project? How many people? Please clarify."

Forgot Turn 1 entirely. Customer must repeat everything. Frustrating. Unprofessional.

✅ With Memory

Turn 1: "I manage ProjectA with 12 people." — stored in memory.

Turn 2: "What is cost per person at 120000 budget?"

Agent: "For your ProjectA team of 12: 120000 ÷ 12 = 10000 INR per person."

Remembered without being told again. Natural professional conversation.

What Should Be Stored?

Facts (names, numbers, decisions), context (what topic we are discussing), history (what was said before), entities (specific things like ProjectA, Ramesh, October budget).

How Long Should It Be Stored?

Short conversation → store everything. Long conversation → store compressed summary. Years of history → store as searchable vectors and retrieve only what is relevant.

The Four Memory Types

TypeStoresBest ForProblemAnalogy
Buffer MemoryEvery message exactly as saidShort chats, 3–10 turnsGets expensive for long chatsCCTV — records every second
Summary MemoryCompressed summary of older messagesMedium chats, 10–50 turns"Keyboard" becomes "electronic device"Meeting minutes — key points only
Entity MemoryFacts about specific named thingsMultiple people and projectsPoor for general conversation flowPhone contacts — facts per person
Vector MemoryAll conversations as searchable vectorsThousands of past chatsComplex setup requiredGoogle search for your memories
ℹ️

Decision guide: Under 10 turns → Buffer. 10–50 turns → Summary. Many named things to track → Entity. Years of history → Vector. For assessments: Buffer Memory is the right choice for short customer service conversations where every detail matters.

Part 14

A2A — Agent to Agent Communication

Analogy — Building a Hospital

One doctor cannot do cardiology, neurology, orthopaedics, and manage the hospital all at once. You need specialists and a coordinator. Cardiologist → heart problems. Neurologist → brain problems. Hospital CEO → coordinates all of them.

Complex AI tasks work the same way: HR Agent (employee expert) + Finance Agent (cost expert) + Report Agent (writing expert) + Orchestrator (coordinates all three). A2A = these agents talking to each other through the orchestrator.

PatternHow It WorksAnalogyBest For
Request-ResponseAgent A asks → Agent B answers → A continues with resultEmail with replySimple data requests between agents
BroadcastOne agent sends to all simultaneously. All work in parallel.WhatsApp group messageCollect data from multiple sources fast
Direct MessageTwo agents talk directly. No orchestrator in the middle.DM between two colleaguesAgents that always work closely together
Event-DrivenEvent fires → subscribed agents react automaticallyDoorbell → you open doorAutomatic reactions to system changes

Safety Rules Between Agents

✓Each agent has only its own tools. HR agent cannot access Finance tools. Like office access cards — you only enter floors you are authorised for.
✓Agents cannot read each other's memory. Separate memory per agent. Shared data only through defined channels.
✓Human approves critical actions. Agent cannot email CFO automatically. Agent cannot delete data. Agent cannot make payments. Always pause. Human reviews. Then proceed.
✓Validate before passing between agents. Bad output from one should not corrupt others. Quality check at every handoff point.

LAB 2 — Build Tools and Test Each Alone

What We Learn
Build tool functions agent will use. Test each alone before connecting to agent. Professional approach: isolate problems early.
Expected Output

Common Mistakes
Tool raises exception → agent crashes entirely. Always return error string. Never raise inside a tool function.
lab2.py
# Normal Python functions
# LLM DECIDES which to call
# Python EXECUTES them
# LLM READS the string result

def check_hr(project):
    try:
        data = {"all": "35 employees across 3 projects",
                "ProjectA": "12 members", "ProjectB": "8 members", "ProjectC": "15 members"}
        return data.get(project, f"{project} not found")
    except Exception as e:
        return f"HR error: {str(e)}"  # never raise — always return string

def check_licenses(month):
    try:
        return f"Software {month}: Microsoft365=45000, Jira=12000, Total=65000 INR"
    except Exception as e:
        return f"License error: {str(e)}"

def check_cloud(month):
    try:
        return f"OCI cloud {month}: Compute=28000, Storage=8000, Total=48000 INR"
    except Exception as e:
        return f"Cloud error: {str(e)}"

def calculate_total(x):
    return "Grand Total: 65000 + 48000 = 113000 INR. Per employee: 3228 INR"

# Test each tool independently before connecting to agent
print("HR all:",       check_hr("all"))
print("HR ProjectA:", check_hr("ProjectA"))
print("Licenses:",    check_licenses("october"))
print("Cloud:",       check_cloud("october"))
print("Total:",       calculate_total(""))
print("All tools working. Ready for LAB 3.")

LAB 3 — Build Agent, See ReAct Live

What We Learn
Connect tools to agent. See Thought/Action/Observation loop live in terminal. verbose=True shows every decision.
Expected Output
Thought → Action: CheckHR → Observation → Thought → Action: CheckLicenses → ... → Final Answer with complete report.
Common Mistakes

lab3.py
from langchain_community.chat_models.oci_generative_ai import ChatOCIGenAI
from langchain.agents import initialize_agent, AgentType
from langchain.tools import Tool

llm = ChatOCIGenAI(
    model_id="cohere.command-a-03-2025",
    service_endpoint="https://inference.generativeai.ap-hyderabad-1.oci.oraclecloud.com",
    compartment_id="ocid1.compartment.oc1..aaaaaaaawsdiuqgqqkxt7wavfbqnqvf2b5kvqmmbghisajmcoqaeqhozbiea",
    auth_type="API_KEY",
    model_kwargs={"temperature": 0.3, "max_tokens": 600}
)

def check_hr(p):
    return {"all":"35 employees","ProjectA":"12 members","ProjectB":"8 members"}.get(p, "not found")
def check_licenses(m): return f"Software {m}: Total=65000 INR"
def check_cloud(m):   return f"OCI cloud {m}: Total=48000 INR"
def calculate_total(x): return "Grand Total=113000 INR. Per employee=3228 INR"

# Description is the most important signal for tool selection
tools = [
    Tool(name="CheckHR",
         description="Get employee count. Input: project name or 'all'.",
         func=check_hr),
    Tool(name="CheckLicenses",
         description="Get software license costs. Input: month like 'october'.",
         func=check_licenses),
    Tool(name="CheckCloud",
         description="Get OCI cloud spending. Input: month like 'october'.",
         func=check_cloud),
    Tool(name="CalculateTotal",
         description="Get grand total. Use after getting all individual costs.",
         func=calculate_total)
]

agent = initialize_agent(
    tools=tools, llm=llm,
    agent=AgentType.ZERO_SHOT_REACT_DESCRIPTION,
    verbose=True,              # shows Thought/Action/Observation live
    max_iterations=8,           # safety: stop after 8 steps maximum
    handle_parsing_errors=True  # bad LLM format: retry not crash
)

print("="*55 + "\nRAMESH TYPES ONE GOAL:\n" + "="*55)
result = agent.invoke({"input": "Get team size, software costs, cloud costs for October. Calculate total."})
print("\n=== FINAL ANSWER ===")
print(result["output"])

Point out to students during demo: Count the steps in the verbose output. Agent decided 4 steps — CheckHR, CheckLicenses, CheckCloud, CalculateTotal. YOU wrote zero steps in the goal. Agent decided all of them. That is the fundamental difference from Chain.

Part 17 — Real OCI Errors

Why Agents Fail — OCI Specific Errors

These are real errors you will see in Cloud Shell. Know them. Fix them quickly. Never be surprised.

oci.exceptions.ServiceError: status 401 — NotAuthenticated
Cause: API key wrong, expired, or revoked. Compartment ID incorrect for this tenancy.
Fix: Check ~/.oci/config file exists and is complete. Verify fingerprint matches OCI Console → Profile → API Keys. Check compartment_id value.
oci.exceptions.ServiceError: status 429 — TooManyRequests
Cause: Too many calls per minute to OCI Gen AI service. Rate limit exceeded.
Fix: import time; time.sleep(60) then retry. Reduce max_iterations to make fewer calls per agent run.
oci.exceptions.ServiceError: status 503 — ServiceUnavailable
Cause: OCI Gen AI service temporarily down or overloaded.
Fix: Wait 2-3 minutes. Check OCI Status page. handle_parsing_errors=True handles some 503 cases automatically.
Agent stopped — max iterations reached without Final Answer
Cause: max_iterations not set, or tool descriptions too vague causing LLM to loop.
Fix: Always set max_iterations=7 or 8. Improve tool descriptions. Add "Do NOT use for X" statements.
Exception in tool function — crashed entire agent
Cause: Tool function raised an exception instead of returning an error string.
Fix: Wrap every tool in try/except. Return error string. Never raise inside a tool. Agent reads string and handles gracefully.
Agent hallucination — numbers not from any tool
Cause: LLM filled gaps from training data when tool did not return the needed info.
Fix: Add to system prompt: "Only use data from tools. If tool did not return it, say you don't know. Never make up numbers."

Part 19 —

Interview Questions

What is the difference between Chain and Agent?
Chain = you define the steps. AI follows your steps in fixed order. Agent = you give a goal. AI defines and executes the steps dynamically at runtime to achieve it.
When NOT to use an Agent?
When steps are fixed and known in advance. Use Chain instead — simpler, faster, more predictable, easier to debug. Agent adds value only when AI needs to decide steps dynamically. Over-engineering with agents adds unnecessary cost and unpredictability.
What is the ReAct pattern and who does each step?
Reasoning + Acting. Thought → Cohere on OCI decides which tool. Action decision → LangChain reads LLM output, finds tool in registry. Execution → Python on Cloud Shell runs the function. Observation → Cohere reads the result. Repeat until done.
How does AI pick the right tool?
Tool description is the most important signal, but not the only one. LLM also considers tool name, user query, system prompt, previous observations, and conversation context. Write clear descriptions including what NOT to use the tool for.
What is the difference between Planning and Reasoning?
Planning = before any tool is called, happens once at start, decides the sequence. Reasoning = after every single tool result, happens multiple times, decides whether to continue or write the final answer. Both happen inside Cohere LLM on OCI.
Who actually executes the Python tool function?
Python on your machine or Cloud Shell. Not the LLM. LLM decides WHICH function to call. LangChain finds it in the tool registry. Python executes it. LLM reads the string result. This separation explains most debugging confusion.
Why return string from tool instead of raising exception?
Exception crashes the entire agent immediately. Error string lets the agent read it as an observation, handle gracefully, and try another approach or give a partial answer. Always wrap tools in try/except. Always return string.
Which memory type for a 5-turn customer service conversation?
ConversationBufferMemory. Short conversation — every detail matters. "Wireless keyboard" must not become "electronic device." Buffer stores everything exactly as said. Summary would compress details. Window memory is unnecessary for only 5 turns.
What is A2A?
Agent to Agent communication. Specialist agents coordinated by an orchestrator. HR Agent + Finance Agent + Report Agent, all coordinated by Orchestrator. Each specialist has only its own tools. Like a hospital with specialists and a CEO who coordinates them all.
What is max_iterations for and why always set it?
Safety net. Stops agent after N steps regardless of whether goal was achieved. Prevents infinite loops where agent keeps calling tools without ever reaching Final Answer. Always set 7–10. No exceptions to this rule.
What is the difference between Tool and API?
API = external service endpoint (OCI Gen AI, a database, LinkedIn). Tool = Python function that wraps an API call. Agent never calls APIs directly — agent calls tools. Inside the tool, you call whatever API or system you need.
Why verbose=True? When to use False?
verbose=True shows every Thought/Action/Observation live in terminal. Essential for learning, debugging, and assessment screenshots. In production: set verbose=False and use structured logging (Lab 7 pattern) instead. Verbose output creates noise in production logs.
What OCI error means authentication failed?
oci.exceptions.ServiceError: status 401 — NotAuthenticated. Check ~/.oci/config file. Verify the fingerprint matches OCI Console → Profile → API Keys. Check compartment_id is correct for your tenancy.
What is the OCI SDK layer and why do students miss it?
The OCI Python SDK (pip install oci) sits between LangChain and OCI Gen AI Service. It handles RSA signing, authentication, and HTTP. ChatOCIGenAI uses it internally. Most diagrams show "LangChain → OCI Gen AI" directly and skip this critical layer.
