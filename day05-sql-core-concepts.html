SQL Execution Flow

SQL is written in one order but executed by the database engine in a completely different logical sequence

1.0

Execution Order — All 8 Steps

The database engine processes clauses in this fixed logical order regardless of how you write them

1FROM
Locate and join tables — forms the initial raw dataset
2WHERE
Filter individual rows — before any grouping or math
3GROUP BY
Bundle rows into groups — based on shared column values
4HAVING
Filter groups — using aggregate functions like COUNT() or SUM()
5SELECT
Extract columns and calculate expressions — aliases are created here
6DISTINCT
Remove duplicate rows — from the selected output
7ORDER BY
Sort the final dataset — ASC or DESC
8FETCH/LIMIT
Truncate to N rows — cuts the bottom of the sorted list
⚠️

Key insight: Column aliases defined in SELECT are NOT available in WHERE or HAVING — those clauses run before SELECT. Always repeat the full expression in WHERE.

1.1

FROM

Locates the target tables and joins them — forms the initial raw dataset

⚡ Execution Step #1

Before filtering or selecting columns, SQL first opens and reads the source table. This is always the very first operation — the entire raw table is made available for subsequent steps.

from_clause.sql
SELECT emp_name FROM employees;
💡 What's happening
  • SQL opens and reads the employees table first, before any filtering or column selection happens
1.2

WHERE

Filters individual records based on a condition — before any grouping

⚡ Execution Step #2

Filters data before any grouping or math happens. SQL discards non-matching rows immediately, reducing the dataset that subsequent steps need to process.

where_clause.sql
SELECT emp_name FROM employees WHERE dept = 'IT';
⚠️

Aggregate functions (COUNT, SUM, AVG) cannot be used in WHERE — they require HAVING, which runs after GROUP BY.

1.3

GROUP BY

Bundles remaining rows into distinct groups based on shared column values

⚡ Execution Step #3

After WHERE filters individual rows, GROUP BY organises the survivors into separate buckets based on shared values. Each bucket becomes one row in the output.

group_by_clause.sql
SELECT dept FROM employees GROUP BY dept;
1.4

HAVING

Filters groups created by GROUP BY — allows aggregate conditions

⚡ Execution Step #4

HAVING is the group-level equivalent of WHERE. It runs after GROUP BY and can use aggregate math like COUNT() or SUM() in its conditions.

having_clause.sql
SELECT dept FROM employees
GROUP BY dept
HAVING COUNT(*) > 5;
💡

WHERE vs HAVING: WHERE filters rows before grouping. HAVING filters groups after grouping. Never use aggregate functions in WHERE.

1.5

SELECT

Extracts specific columns and processes mathematical expressions and functions

⚡ Execution Step #5

Column aliases are created here — which is why they cannot be referenced in WHERE or HAVING. Mathematical expressions and function calls are evaluated at this step.

select_clause.sql
SELECT emp_name, (salary * 0.1) AS bonus FROM employees;
1.6

DISTINCT

Eliminates duplicate rows from the final selected output

⚡ Execution Step #6

After SELECT computes all columns, DISTINCT compares entire rows and discards identical duplicates. This requires a sorting pass which adds processing overhead on large datasets.

distinct_clause.sql
SELECT DISTINCT location FROM departments;
1.7

ORDER BY

Sorts the final dataset by specified columns — ASC or DESC

⚡ Execution Step #7

The final, filtered and selected data is sorted for display. Column aliases defined in SELECT are available here since ORDER BY runs after SELECT.

order_by_clause.sql
SELECT emp_name, salary FROM employees
ORDER BY salary DESC;
1.8

FETCH / LIMIT

Truncates the result to a maximum number of rows — cuts the bottom of the sorted list

⚡ Execution Step #8

After sorting, the result is cut down to the specified row count. Always pair with ORDER BY — without it, the rows returned are non-deterministic.

fetch_limit.sql
-- Oracle 12c+ (ANSI standard)
SELECT emp_name, salary FROM employees
ORDER BY salary DESC
FETCH FIRST 3 ROWS ONLY;

-- Older Oracle (pre-12c)
SELECT * FROM (
    SELECT emp_name, salary FROM employees
    ORDER BY salary DESC
) WHERE ROWNUM <= 3;
💡

Oracle uses FETCH FIRST N ROWS ONLY (not LIMIT). LIMIT is MySQL/PostgreSQL syntax and will cause an error in Oracle SQL.

Joins

Link records across distinct tables using shared columns — INNER, LEFT, RIGHT, FULL, CROSS, SELF

2.0

Why Joins?

Normalization splits data into narrow tables — joins stitch them back together at query time

Normalization slices data into narrow, efficient tables to avoid redundancy. Joins stitch these tables back together during a query to answer complex business questions.

💡 Real-world Example
  • To see an employee's name and their department name, you must join the employees table with the departments table using a shared dept_id column
2.1

INNER JOIN

Returns rows only when a matching key exists in BOTH tables — non-matches are dropped

inner_join.sql
SELECT e.emp_name, d.dept_name
FROM   employees e
INNER JOIN departments d ON e.dept_id = d.dept_id;
Output
EMP_NAMEDEPT_NAME
AliceIT
CharlieFinance
💡

Bob has no department and Marketing has no employees — both are excluded from the result.

2.2

LEFT OUTER JOIN

Returns ALL rows from the left table — unmatched right side fills with NULL

left_join.sql
SELECT e.emp_name, d.dept_name
FROM   employees e
LEFT JOIN departments d ON e.dept_id = d.dept_id;
Output
EMP_NAMEDEPT_NAME
AliceIT
BobNULL
CharlieFinance
💡

Bob appears with NULL — he is a new hire not yet assigned to a department.

2.3

RIGHT OUTER JOIN

Returns ALL rows from the right table — unmatched left side fills with NULL

right_join.sql
SELECT e.emp_name, d.dept_name
FROM   employees e
RIGHT JOIN departments d ON e.dept_id = d.dept_id;
Output
EMP_NAMEDEPT_NAME
AliceIT
CharlieFinance
NULLMarketing
💡

Marketing appears with NULL — it is a newly created department with no employees yet.

2.4

FULL OUTER JOIN

Returns ALL rows from both tables — missing matches on either side fill with NULL

full_outer_join.sql
SELECT e.emp_name, d.dept_name
FROM   employees e
FULL OUTER JOIN departments d ON e.dept_id = d.dept_id;
Output
EMP_NAMEDEPT_NAME
AliceIT
BobNULL
CharlieFinance
NULLMarketing
💡 Use Case
  • A master audit list showing all employees (even without departments) AND all departments (even without employees)
2.5

CROSS JOIN

Cartesian product — every row in the first table paired with every row in the second

No ON condition is used. If the first table has 3 rows and the second has 2 rows, the result has 3 × 2 = 6 rows.

cross_join.sql
SELECT s.size_name, c.color_name
FROM   sizes s
CROSS JOIN colors c;
Output (3 sizes × 2 colors = 6 rows)
SIZE_NAMECOLOR_NAME
SmallRed
SmallBlue
MediumRed
MediumBlue
LargeRed
LargeBlue
2.6

SELF JOIN

Joins a table to itself using two aliases — compares rows within the same table

self_join.sql
SELECT emp.emp_name  AS worker,
       mgr.emp_name  AS supervisor
FROM   employees emp
LEFT JOIN employees mgr ON emp.manager_id = mgr.emp_id;
Output
WORKERSUPERVISOR
AliceCharlie
BobCharlie
CharlieNULL
💡

Charlie has no supervisor (NULL) because he is the top-level manager — his manager_id is NULL so the LEFT JOIN finds no match.

🗂️

Set Operators

Stack rows from multiple queries vertically — unlike joins which combine columns side-by-side

3.0

Rules for Set Operators

📋

Both queries must return the same number of columns with compatible data types. Column names in the result come from the first query.

OperatorDuplicatesSort PassPerformance
UNIONRemovedYesSlower
UNION ALLKeptNoFaster
INTERSECTRemovedYesMedium
MINUSRemovedYesMedium
3.1

UNION

Merges rows from two queries and removes all duplicates

union.sql
SELECT name FROM active_employees
UNION
SELECT name FROM retired_employees;
A clean, unique list of names — duplicates removed, sorted alphabetically.
3.2

UNION ALL

Merges rows quickly without sorting — keeps all duplicates exactly as they appear

union_all.sql
SELECT user_id FROM server1_logs
UNION ALL
SELECT user_id FROM server2_logs;
All IDs from both servers — duplicates retained exactly as they appear in the source data.
⚡

Always prefer UNION ALL over UNION when duplicates are acceptable or you know none exist — it skips the expensive deduplication sort pass.

3.3

INTERSECT

Returns only distinct rows that appear in BOTH query results simultaneously

intersect.sql
SELECT email FROM customers
INTERSECT
SELECT email FROM suppliers;
A list of emails belonging to people who are registered as both customers and suppliers.
3.4

MINUS (EXCEPT)

Returns rows from the first query that do NOT exist in the second — subtracts matching rows

minus.sql
SELECT emp_id FROM employees
MINUS
SELECT emp_id FROM expense_reports;
Employee IDs who have zero expense records — appear in employees but not in expense_reports.
💡

Oracle uses MINUS. PostgreSQL and SQL Server use EXCEPT. Both produce the same result.

🔍

Subqueries

An inner query nested inside a parent statement — Single Row, Multiple Row, Nested, Correlated, IN, EXISTS, ANY, ALL

4.1

Single Row Subquery

Returns exactly one column and one row — used with =, >, < operators

single_row_sub.sql
-- Employees earning more than company average
SELECT emp_name FROM employees
WHERE  salary > (SELECT AVG(salary) FROM employees);
💡

The inner query runs first, returns one value (e.g., 50000), and the outer query compares each row's salary against that value.

4.2

Multiple Row Subquery

Returns a list of multiple values — used with IN, ANY, or ALL operators

multi_row_sub.sql
-- Employees in New York departments
SELECT emp_name FROM employees
WHERE  dept_id IN (
    SELECT dept_id FROM departments
    WHERE  location = 'New York'
);
4.3

Nested Subquery

A subquery containing another subquery inside itself — multiple layers of nesting

nested_sub.sql
-- Employees in the dept with the highest average salary
SELECT emp_name FROM employees
WHERE  dept_id = (
    SELECT dept_id FROM employees
    GROUP BY dept_id
    HAVING AVG(salary) = (
        SELECT MAX(AVG(salary))
        FROM   employees
        GROUP BY dept_id
    )
);
4.4

Correlated Subquery

References a column from the outer query — re-executes once for every outer row

correlated_sub.sql
-- Employees earning more than their own dept's average
SELECT emp_name, salary, dept_id
FROM   employees outer_emp
WHERE  salary > (
    SELECT AVG(salary)
    FROM   employees inner_emp
    WHERE  inner_emp.dept_id = outer_emp.dept_id  -- outer reference
);
⚠️

Correlated subqueries re-execute for every outer row — they can be slow on large tables. Consider rewriting as a JOIN with a derived table for better performance.

4.5

IN Operator

Evaluates to TRUE if the value matches any entry in the returned subquery list

in_operator.sql
-- Employees in departments 10, 20, or 30
SELECT emp_name FROM employees
WHERE  dept_id IN (10, 20, 30);
4.6

EXISTS Operator

Returns TRUE if the subquery yields at least one row — halts immediately on first match

exists_operator.sql
-- Departments that have at least one employee
SELECT dept_name FROM departments d
WHERE  EXISTS (
    SELECT 1 FROM employees e
    WHERE  e.dept_id = d.dept_id
);
⚡

EXISTS vs IN: EXISTS stops at the first match (highly efficient for large subqueries). IN builds the full result set first then checks. Use EXISTS for existence checks.

4.7

ANY Operator

TRUE if the value satisfies the condition against at least one element in the list

any_operator.sql
-- Employees earning more than the lowest-paid Sales worker
SELECT emp_name FROM employees
WHERE  salary > ANY (
    SELECT salary FROM employees
    WHERE  dept = 'Sales'
);
💡

ANY = "more than at least one" = more than the minimum. Equivalent to > MIN(...).

4.8

ALL Operator

TRUE only if the value satisfies the condition against every element in the list

all_operator.sql
-- Employees earning more than the highest-paid Sales worker
SELECT emp_name FROM employees
WHERE  salary > ALL (
    SELECT salary FROM employees
    WHERE  dept = 'Sales'
);
💡

ALL = "more than every single one" = more than the maximum. Equivalent to > MAX(...).

🪟

Views

Virtual tables based on stored SQL queries — do not store physical data rows

5.1

Simple View

Single base table, no functions or groupings — supports INSERT and UPDATE through the view

simple_view.sql
-- A safe window that hides salary details
CREATE VIEW v_emp_public AS
SELECT emp_id, emp_name, dept_id
FROM   employees;
✅

Simple views on a single table without functions, DISTINCT, or GROUP BY are updatable — you can INSERT or UPDATE rows through them.

5.2

Complex View

Contains joins, aggregates, or GROUP BY — strictly read-only

complex_view.sql
-- Department salary summary (read-only)
CREATE VIEW v_dept_summary AS
SELECT   dept_id,
         COUNT(*) AS staff_count,
         SUM(salary) AS payroll
FROM     employees
GROUP BY dept_id;
⚠️

Views with joins, GROUP BY, aggregate functions, DISTINCT, or ROWNUM are read-only — DML operations through them are not permitted.

5.3

CREATE VIEW

Generates a new saved virtual table definition inside the database schema

create_view.sql
CREATE VIEW v_it_staff AS
SELECT emp_name
FROM   employees
WHERE  dept_id = 10;
5.4

CREATE OR REPLACE VIEW

Modifies an existing view without dropping it — preserves granted permissions

replace_view.sql
-- Adds email column to existing IT staff view
CREATE OR REPLACE VIEW v_it_staff AS
SELECT emp_name, email
FROM   employees
WHERE  dept_id = 10;
✅

Always use CREATE OR REPLACE instead of DROP + CREATE — it preserves all GRANT permissions attached to the existing view.

5.5

DROP VIEW

Permanently removes the view definition — does NOT delete data from underlying tables

drop_view.sql
DROP VIEW v_it_staff;
💡

Dropping a view only removes the stored query definition. All data in the underlying base tables remains completely unaffected.

📑

Indexes

Pointer structures that map column values to physical disk locations — like an index at the back of a book

6.0

Why Index?

Without an index, finding one row requires scanning every row on disk — Full Table Scan

Without an index, finding a single row requires the database engine to read every single row on disk sequentially (Full Table Scan). An index allows the engine to jump directly to the exact row, shifting performance from slow O(n) to near-instant O(log n).

💡 Real-world Example
  • Searching for a specific customer by phone number out of 1,000,000 records — without an index, every row is read. With an index, Oracle jumps straight to the matching entry.
6.1

CREATE INDEX

Standard B-Tree index — speeds up searches using =, >, <, BETWEEN conditions

create_index.sql
-- Index for rapid profile searches by last name
CREATE INDEX idx_emp_lastname ON employees(last_name);
6.2

UNIQUE INDEX

Fast retrieval AND enforces uniqueness — no two rows can share the same value

unique_index.sql
-- Fast lookup + data integrity for national ID
CREATE UNIQUE INDEX idx_gov_id ON employees(government_id);
6.3

Composite Index

Index across two or more columns — speeds up multi-column search conditions

composite_index.sql
-- Dashboard always filters by dept AND job title together
CREATE INDEX idx_dept_job ON employees(dept_id, job_title);
6.4

Index Usage Rules

Column order matters — the database auto-selects when to use an index based on your WHERE clause

Databases automatically decide when to use an index based on the structure of your WHERE clauses. The order of columns in composite indexes matters significantly.

Query PatternIndex (dept_id, job_title)Used?
WHERE dept_id = 10Searches first column✅ Yes
WHERE dept_id = 10 AND job_title = 'Dev'Searches both columns✅ Yes
WHERE job_title = 'Dev'Skips leading column❌ No
⚠️

An index on (dept_id, job_title) will help queries filtering by dept_id alone, but will NOT help queries filtering by job_title alone — always put the most selective column first.

🚀

Query Optimization

Writing SQL that is both correct AND efficient — without exhausting system resources

7.1

Full Table Scan

Database reads every single page of the table — highly inefficient on large tables

Happens when you query without indexes or filter on a non-indexed column. Every row on disk is read even if only one row matches. For small tables this is fine; for millions of rows it becomes a serious bottleneck.

🔴

When it occurs: No index on the filtered column, function wrapped around an indexed column in WHERE, or the optimizer estimates a full scan is cheaper (e.g., returning >20% of rows).

7.2

Index Scan

Database navigates the compact index tree to pinpoint rows directly — bypasses bulk of raw table

The engine hits the index tree first (O(log n) traversal), then fetches only the matching rows from disk. Dramatically faster than full table scan for selective queries.

💡 Example
  • Looking up a primary key ID value — Oracle traverses the B-Tree index in ~20 steps regardless of table size, then reads exactly 1 row
7.3

Avoid SELECT *

Selecting all columns forces unnecessary disk I/O and can break index-only processing

❌ bad_practice.sql
-- Reads ALL 50 columns
SELECT * FROM employees;
✅ optimized.sql
-- Reads only needed columns
SELECT emp_name, email
FROM   employees;
✅

Listing only the columns you need reduces disk I/O, network transfer size, and enables index-only scans where all needed data is in the index itself.

7.4

Efficient WHERE Clauses

Keep columns raw on their side of the operator — wrapping columns in functions disables index matching

❌ disables_index.sql
-- EXTRACT() wraps the column
-- index on hire_date is disabled
SELECT emp_name FROM employees
WHERE  EXTRACT(YEAR FROM hire_date)
       = 2026;
✅ uses_index.sql
-- Column is raw — index is used
SELECT emp_name FROM employees
WHERE  hire_date >= '2026-01-01'
  AND  hire_date <= '2026-12-31';
7.5

Join Optimization

Ensure linking columns share identical data types and both sides have indexes

Joining an INT column to a VARCHAR column forces the database to convert data types on the fly for every single row, causing severe slowdowns. Always ensure ON clause columns are the same data type and indexed on both tables.

🔴

Type mismatch: Joining employees.dept_id (NUMBER) to departments.dept_id (VARCHAR2) silently disables the index and scans every row on at least one side.

✅

Always index foreign key columns used in JOIN ON conditions. This is one of the highest-impact optimizations available.

7.6

Query Tuning Basics

Use EXPLAIN PLAN to visualise Oracle's internal execution roadmap and find bottlenecks

EXPLAIN PLAN shows you exactly which operations Oracle will perform — index scans, full table scans, join methods — without actually running the query. It reveals where bottlenecks are hiding.

explain_plan.sql
-- Step 1: Generate the plan (no data returned)
EXPLAIN PLAN FOR
SELECT emp_name, salary
FROM   employees
WHERE  department_id = 10;

-- Step 2: Display the plan
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
💡

Look for FULL TABLE SCAN in the plan output — that is the operation to eliminate. INDEX RANGE SCAN or INDEX UNIQUE SCAN are what you want to see.
