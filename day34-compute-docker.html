Deep Dive

OCI Functions — Serverless

Serverless means you write code and OCI handles everything else — the server, OS, scaling, patching. You only pay when your code actually runs.

Analogy — Taxi vs Personal Car

Personal Car (Compute VM): Always available in your garage. You pay insurance, fuel, and maintenance 24/7 — even when the car sits parked and unused.

Ola/Uber (OCI Functions): Call when you need it. Arrives in seconds. Pay only for the ride. When you're not traveling, you pay nothing.

Ramesh's monthly report runs once a month for 10 minutes. Functions: pay for 10 minutes. VM: pay for 720 hours (full month). Functions is 4,320× cheaper for this use case.

✅ Use OCI Functions When

Event-driven tasks triggered by file upload or API call. Infrequent tasks: monthly reports, weekly data sync. Short duration: under 5 minutes. Variable load: sometimes 0 calls, sometimes 1,000. Microservices where each function does one thing.

❌ Do NOT Use Functions When

Task runs longer than 5 minutes (cold start delays). Always-on services that must respond instantly (use Container Instance). Needs GPU compute (not supported). Persistent connections required. Complex stateful processing.

Deep Dive

OCI Container Instances — Always-On

Run a Docker container directly on OCI without Kubernetes. The container is always running — like a mini server you don't have to manage.

Key Characteristics

Always running — no cold start delay. Instant response to every API call. Pay by the hour whether used or not. Deploy with one image URL. Environment variables for config. IAM for security.

✅ Use Container Instances When

Agent must respond instantly — RAG Q&A cannot have 3-second cold start. Always-on REST API accessible by APEX. Medium traffic with steady load. Simple single-container deployment without Kubernetes complexity.

💡

Functions vs Container Instances for RAG Q&A: OCI Functions has a cold start of 1–3 seconds — the container must start before running your code. For an employee typing a question, that 3-second delay is unacceptable. Container Instance is always running — employee's question is answered instantly. Use Container Instance for RAG Q&A. Use Functions for ingestion and report generation where a brief delay is acceptable.

Full Control

Compute VMs — When You Need Complete Control

✅ Use Compute VM When

Full OS control needed. Install custom software not available in containers. Long-running 24/7 background processes. Persistent disk storage for stateful data like ChromaDB with persist_directory. Legacy applications that cannot be containerised easily.

❌ Do NOT Use VM When

Simple container deployment exists (use Container Instance — simpler). Infrequent tasks (use Functions — cheaper). No DevOps team to manage OS patching and upgrades. Auto-scaling is required (use OKE).

Architecture Principle

Stateless Compute vs Stateful Data

🔑

The golden rule of cloud architecture: Keep your compute stateless. Keep your data in stateful services. This is why Functions and Container Instances can scale — any instance can serve any request because state lives in the database, not in the compute.

Stateless — Compute

Does NOT remember previous calls. Every invocation starts fresh — clean slate. No memory between calls. OCI Functions: stateless. Container Instances: stateless. This is intentional — enables horizontal scaling.

Examples: your agent code, API handlers, ingestion pipeline

Stateful — Data Stores

Remembers data between calls. Data persists when compute stops. Multiple compute instances all read the same data. ChromaDB: stateful. OCI Object Storage: stateful. This is where the "memory" lives.

Examples: ChromaDB (vectors), Object Storage (PDFs), databases

Analogy — Waiter and Kitchen

Stateless waiter: A new customer arrives. The waiter has no memory of any previous customer. Takes each order fresh. The waiter can be replaced by any other waiter — all waiters are interchangeable.

Stateful kitchen: Recipes stored in the kitchen. Ingredients in the pantry. Any waiter can serve from the same kitchen. The kitchen holds all the knowledge.

In LTIMindtree's RAG system: The agent (waiter) is stateless — any container instance handles any request. ChromaDB and Object Storage (kitchen) are stateful — store all vectors and documents. 10 Container Instances can run in parallel, all reading from the same ChromaDB. Results are identical regardless of which instance answers.

The Core Problem

Why Containerise AI Agents?

Without Docker — "Works on My Machine"

Cloud Shell: Python 3.11, langchain 0.3.30, chromadb 0.5.0. OCI VM: Python 3.8, no langchain installed.

ImportError: No module named 'langchain'

Same code. Different environment. Different result. Breaks in production.

With Docker — "Works Everywhere"

Package together: Python 3.11 (exact version) + all packages at exact versions + your agent code + configuration.

Same container → same behaviour on Cloud Shell, OCI Container Instance, OKE, your Mac. The environment is the container.

Analogy — Shipping Container

Before shipping containers (1950s): Ships goods loose. Different boxes, different sizes, different handling at every port. Loading takes days. Goods get damaged or lost. Each port needs different equipment.

After shipping containers: Everything packed in a standardised container. Same container → same process at Rotterdam, Dubai, Singapore, Mumbai. Same crane, same truck, same process. Fast. Reliable. Predictable.

Docker container = shipping container for software. Your agent packaged once → runs identically at every destination. Cloud Shell, OCI server, or your colleague's laptop — all identical.

Fundamental Concepts

Dockerfile vs Image vs Container

The most common point of confusion for beginners. Three completely different things that work together.

📋
Dockerfile
= Recipe Card
Written instructions for HOW to build the environment. Not runnable by itself. Tells Docker: use Python 3.11, install these packages, copy this code. One Dockerfile → creates the image.
📦
Docker Image
= Prepared Meal
The Dockerfile was executed. Result: a packaged, ready-to-run snapshot of the entire environment. Stored in OCIR. Not yet running. Can create many containers from one image.
🚀
Docker Container
= Meal Being Served
The image is now running. An active process on a server. Consuming CPU and memory. Accepting requests. Can stop, start, pause. Many containers can run from one image simultaneously.
🍽️

Restaurant analogy complete: Recipe card (Dockerfile) → Prepared dish (Image) → Dish served on table (Container). One recipe → many prepared dishes. One image → many running containers. Modifying the recipe means creating a new dish — you don't modify the already-prepared dish.

Writing Code

The Dockerfile — Line by Line

🐳 Dockerfile — LTIMindtree Agent
# ── LINE 1: BASE IMAGE ───────────────────────────────────
FROM python:3.11-slim
python:3.11-slim = Python 3.11 + minimal Linux OS. "slim" saves ~200MB vs full image.
This ensures Python 3.11 everywhere — Cloud Shell, OCI, your Mac — always identical.

# ── LINE 2: WORKING DIRECTORY ────────────────────────────
WORKDIR /app
/app is the folder INSIDE the container. NOT your project folder name on your machine.
All subsequent commands run from /app. Standard convention — use /app always.

# ── LINE 3: COPY REQUIREMENTS FIRST ──────────────────────
COPY requirements.txt .
Copy requirements.txt BEFORE your code files. Docker caches each step as a layer.
If requirements.txt unchanged → pip install uses cache → rebuild in seconds not minutes.

# ── LINE 4: INSTALL PACKAGES ──────────────────────────────
RUN pip install --no-cache-dir -r requirements.txt
--no-cache-dir: don't save pip's download cache inside the image → smaller image size.

# ── LINE 5: COPY ALL PROJECT FILES ───────────────────────
COPY . .
Copy AFTER pip install. If only code changes (not requirements.txt), Docker skips pip step.
Much faster rebuilds during development when you're only changing Python code.

# ── LINE 6: RUN COMMAND ───────────────────────────────────
CMD ["python", "main.py"]
What executes when the container starts. python main.py = run your agent.
Using ["python", "main.py"] (exec form) is preferred over "python main.py" (shell form).

The .dockerignore File — Critical for Security

.dockerignore — files excluded from Docker image
# NEVER put credentials in Docker image
.oci/              # ← OCI private keys — NEVER in image!
.env               # ← local environment variables

# Python clutter
__pycache__/
*.pyc

# Version control
.git/

# Local virtual environment
venv/
🚨

Security critical: Without .dockerignore, docker build copies your .oci/ folder (containing your OCI private key) into the image. When you push to OCIR, any developer with pull access can extract your private key and access your entire OCI account. Always add .dockerignore before your first docker build.

Security

Credentials — Three Ways to Authenticate

Dev — Volume Mount

Mount your local .oci folder into the container at runtime. Container reads credentials from /root/.oci/config. Good for development and Cloud Shell.

docker run -v ~/.oci:/root/.oci agents

Test — Env Variables

Pass values as -e flags at runtime. Good for CI/CD pipelines. Secrets managed outside code. Different values for dev, test, and production environments.

docker run -e OCI_COMPARTMENT_ID=... agents

Prod — Instance Principal ✅

OCI resource has its own identity. No keys in code or container. Most secure. Agent authenticates as the compute instance itself using OCI IAM automatically.

signer = InstancePrincipalsSecurityTokenSigner()

Image Registry

OCI Container Registry (OCIR) Workflow

🗄️

OCIR = OCI Container Registry — private Docker image registry inside OCI. Like Docker Hub but private, faster (same OCI network as Container Instance), IAM-controlled, and data never leaves OCI. Always push to OCIR instead of Docker Hub for OCI deployments.

OCIR Image URL Format

OCIR image URL anatomy
ap-hyderabad-1.ocir.io / ocigenai58 / ltimindtree-agent : v1
│                        │             │                  │
│                        │             │                  └── version tag
│                        │             └── image name
│                        └── namespace (your tenancy)
└── OCI region registry endpoint

Complete OCIR Workflow — Commands

1
Build the Docker image locally
docker build -t ltimindtree-agent .
The . means "use current folder for Dockerfile and all files".
2
Tag with full OCIR address
docker tag ltimindtree-agent ap-hyderabad-1.ocir.io/ocigenai58/ltimindtree-agent:v1
Gives the local image its full registry path for pushing.
3
Get Auth Token from OCI Console
OCI Console → Profile → My Profile → Auth Tokens → Generate Token. Copy immediately — shown only once. This is the password for OCIR login.
4
Login to OCIR
docker login ap-hyderabad-1.ocir.io --username ocigenai58/ragulrajdhamodharan@gmail.com
Enter Auth Token as password when prompted.
5
Push image to OCIR
docker push ap-hyderabad-1.ocir.io/ocigenai58/ltimindtree-agent:v1
Image is now stored in OCI. Any OCI service can pull it.
6
Deploy from OCIR
OCI Console → Developer Services → Container Instances → Create → use OCIR image URL. Or OCI Functions → deploy from container image.

    tools=[Tool(name="BudgetTool", func=check_budget,

orchestrator = initialize_agent(

        Tool(name="AskFinanceAgent", func=ask_finance,

Interview Preparation

Interview Questions — Beginner to Advanced

Beginner Level

Why is Cloud Shell not suitable for production AI agents?
Cloud Shell expires after 20 minutes of idle time — the agent stops automatically. It is personal (only your OCI account), so CFO or HR team cannot trigger it. There is no REST endpoint — APEX cannot call it via URL. There is no monitoring, alerting, or centralised logging. Production needs a permanent, shared, monitored service — which requires OCI Compute (Functions, Container Instances, or VMs).
What is the difference between a Dockerfile, Docker Image, and Docker Container?
Dockerfile is the recipe — written instructions for building the environment (base image, packages, code). Image is the prepared meal — Dockerfile was executed and result is a packaged, ready-to-run snapshot stored in OCIR. Container is the meal being served — the image is running as an active process on a server. One Dockerfile builds one image. One image can run as many containers simultaneously. You cannot change a running container's image — you modify the Dockerfile, rebuild the image, and redeploy.
What is OCIR and why use it instead of Docker Hub?
OCIR is OCI Container Registry — a private Docker image registry inside OCI. Advantages over Docker Hub: faster image pulls (same OCI network as Container Instance, no internet transfer), IAM-controlled access (only authorised OCI users can push/pull), data never leaves OCI network (compliance), integrated with OCI services. Always use OCIR for OCI deployments instead of Docker Hub.

Intermediate Level

OCI Functions vs Container Instances — when to use each for a RAG system?
OCI Functions for ingestion (triggered by file upload event, runs 2–3 minutes, stops — pay per call, very cheap for infrequent task) and for report generation (triggered monthly by scheduler, runs 10 minutes). Container Instance for the RAG Q&A API (must respond to employee questions instantly — Functions has 1–3 second cold start which is unacceptable for interactive Q&A, Container Instance is always running with instant response). The key distinction: cold start matters for user-facing APIs but not for background processing.
Why put .oci/ in .dockerignore? What happens if you forget?
The .oci/ folder contains your OCI private key (oci_api_key.pem) and config file. If you forget .dockerignore, docker build copies this folder into the image layer. When you push to OCIR, anyone with image pull access can extract the image layers and read your private key. With that key, they can authenticate as you and access your entire OCI account, including all resources, data, and services. Always add .dockerignore before your first docker build and verify with docker history that no sensitive files are in the image layers.
What is Instance Principal authentication and why is it the best for production?
Instance Principal lets an OCI compute resource (Function, Container Instance, VM) authenticate to other OCI services using its own identity — no API keys required. The OCI SDK detects it automatically at runtime. Benefits: no keys to rotate or manage, no risk of key exposure in code or environment variables, automatic credential refresh, and audit logs show the compute resource as the actor rather than a user. In production, use: signer = oci.auth.signers.InstancePrincipalsSecurityTokenSigner() and pass it to OCI SDK clients instead of reading from config file.

Advanced Level

Explain the stateless compute + stateful data architecture principle.
Stateless compute (Functions, Container Instances) does not retain any state between invocations — each call starts fresh. This enables horizontal scaling: add more instances, each handles any request independently. Stateful data (ChromaDB, Object Storage, databases) persists between calls and is shared by all compute instances. The design rule: all shared state lives in managed data services, never in compute. Result: 10 Container Instances can run in parallel, all reading from the same ChromaDB. Any instance handles any request. If one fails, others continue. This is the foundation of cloud-native, scalable AI systems.
Why does Dockerfile copy requirements.txt before code files? What benefit does this provide?
Docker builds images as a series of layers, and it caches each layer. If a layer's input hasn't changed, Docker reuses the cached result instead of rebuilding. By copying requirements.txt first and running pip install before copying code files, we create a cache-friendly layer order: when you only change Python code (agents.py, main.py), Docker's cache hit on the pip install layer means packages are not reinstalled — rebuild takes seconds instead of minutes. If you copy all files first, any code change invalidates the pip layer cache and forces a full package reinstall on every rebuild.
What is CI/CD and how does it connect to Docker and OCI?
CI/CD = Continuous Integration/Continuous Deployment. A pipeline that automates everything from code commit to production deployment. Developer pushes code to GitHub → GitHub Actions (or OCI DevOps) is triggered → runs automated tests → builds new Docker image → pushes to OCIR → updates Container Instance or deploys new OCI Function version. Without CI/CD: 10 manual steps, 30 minutes, human error risk. With CI/CD: automatic, 5 minutes, consistent, repeatable. For AI agents in production, CI/CD ensures every code change is tested and deployed safely, and a record exists of every deployment and its contents.

Day 39 Summary

☁️ OCI Compute Options
  • Functions = serverless, event-driven, pay per call
  • Container Instance = always-on, instant response, per hour
  • OKE = enterprise scale, auto-scaling, microservices
  • VM = full OS control, persistent disk
  • Functions for ingestion + report generation
  • Container Instance for RAG Q&A API
  • Functions cold start = 1–3 sec (unacceptable for Q&A)
  • Container Instance = instant (no cold start)
🐳 Docker Essentials
  • Dockerfile = recipe → Image = prepared meal → Container = serving
  • One image → many containers running simultaneously
  • Copy requirements.txt BEFORE code → faster rebuilds
  • Always create .dockerignore — exclude .oci/ and .env
  • --no-cache-dir → smaller image size
  • CMD ["python", "main.py"] → exec form (not shell form)
  • docker build -t name . → . is mandatory (current folder)
  • Same image → same behaviour everywhere
🔐 Security & Architecture
  • Dev: volume mount (-v ~/.oci:/root/.oci)
  • Prod: Instance Principal (no keys in container)
  • OCIR = private registry inside OCI (not Docker Hub)
  • Stateless compute + stateful data = scalable design
  • IAM Dynamic Group for agent access to OCI services
  • Private subnet for container — API Gateway in front
  • OCI Logging + Monitoring + Alarms = full observability
  • CI/CD: GitHub → Actions → OCIR → OCI deploy
⚠️ Common Mistakes
  • Forgetting .dockerignore → credentials in image
  • Using Cloud Shell for production → sessions expire
  • Forgetting . in docker build → "no context" error
  • Using Docker Hub instead of OCIR for OCI deployments
  • Choosing Container Instance for monthly reports → expensive
  • Choosing Functions for Q&A API → cold start delays
  • Hardcoding secrets in Dockerfile or Python files
  • Not tagging images with versions — using only "latest"
