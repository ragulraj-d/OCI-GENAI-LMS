Basic Networking — One Concept

Part B · REST, JSON & HTTP — Foundation for Gen AI
🔗
REST APIs, HTTP Methods, Headers & JSON
Bridge
Why This Matters Right Now
From Day 33 onwards, your APEX app calls OCI Gen AI via a REST API. The request is JSON. The response is JSON. APEX parses that JSON and shows the AI answer on screen. Understanding REST and JSON now means you will not be confused when the code uses apex_web_service.make_rest_request or apex_json.get_varchar2.
HTTP MethodPurposeReal Example
GETFetch / Read dataFetch employee list from HR system
POSTSend / Create dataSubmit leave request, call Gen AI API
PUTUpdate existing dataUpdate employee salary
DELETERemove dataDelete a cancelled leave request
Status CodeMeaning
200Success — data returned correctly
201Created — new record made
400Bad Request — wrong JSON format
401Unauthorized — missing or invalid API key
404Not Found — wrong endpoint URL
500Server Error — something failed server-side
Request Headers
Every HTTP request carries headers — metadata about the request itself. 
-- Common headers used with OCI Gen AI:
Content-Type: application/json   -- "I am sending JSON"
Accept: application/json         -- "I want JSON back"
Authorization: Bearer <token>   -- "Here is my identity proof"
In APEX, when you use a Web Credential with OCI Signing, headers are handled automatically. Your private key signs every request so OCI can verify it came from you.
JSON — Parse vs Serialize
Serialize = Convert your data INTO JSON to send.  Parse = Convert JSON response BACK into usable values. 
-- Serialize: build request body
l_body := '{"message":"How many leaves do I have?","maxTokens":500}';

-- Parse: extract value from response
apex_json.parse(l_response);
l_answer := apex_json.get_varchar2(p_path => 'chatResponse.text');
JSON Types: String "text" · Number 25 · Boolean true/false · Array [1,2,3] · Object {key:value} · Null null
Authentication Types
API Key — Simple key in header  |  Bearer Token — JWT, expires after a time window  |  OCI Signing — Private-key signing (used for all OCI APIs)  |  Basic Auth — Username:Password encoded in header
Part C · Oracle APEX — The Stack
🏗
Oracle Stack — DB → ORDS → APEX
Core Concept
The Business Problem APEX Solves
A company with 5,000 employees gets 200 leave requests every Monday. HR manages them on Excel and WhatsApp. The HR manager spends 3 hours sorting. Approvals go on email — no tracking. Employees have no visibility.

This is a real problem in every large organisation. APEX solves it in 2 days: one developer, one database, one application — leave request → manager approval → employee notification — all in a browser, no installation, no separate web server.
What is Oracle APEX
APEX (Application Express) is a low-code web application platform that runs inside Oracle Database. Build complete enterprise web apps using only SQL — no separate HTML, CSS, or JavaScript required. Used by 50,000+ organisations worldwide.

ORDS — Middleware. Converts HTTP to DB calls. Manages a connection pool — pre-opened DB connections shared across all requests (much faster than opening a new connection each time).

APEX Engine — Reads your page definition from the DB and dynamically builds HTML. You never write HTML — APEX generates it.

Part D · Session Management
🔛
Sessions — What Happens After Login
Core Concept
Why Sessions Exist
HTTP is stateless — every request is independent. The server has no memory of who you are between requests. Sessions solve this: after login, APEX creates a session that identifies you for your entire visit. Without sessions, you would need to log in on every single click.
Session Lifecycle
User Logs In
→
Session Created
→
Session ID Generated
→
ID in URL / Cookie
→
User Works
→
Timeout / Logout
ConceptWhat It IsWhere You See It
Session IDUnique number identifying your loginThird segment in URL: f?p=100:1:12345678
Session StateAll item values stored server-side for your sessionValues persist as you navigate between pages
Session TimeoutAuto-logout after inactivityApp Properties → Security → Session Timeout
LogoutClears session state and session IDNavigation bar logout link
Real Example
You open Employee Edit form (Page 3). You select Department = Sales. You navigate to Page 4. You return to Page 3. The Department field still shows Sales — APEX stored it in Session State. Session State is APEX's short-term memory for your active session.
Security Note
Always set session timeout for apps handling sensitive data. Recommended: 30 minutes for internal HR and finance apps. This prevents others accessing an unattended open session.

Part H · List of Values (LOV)
📋
LOV — Used in Almost Every APEX Form
Core Component
What is an LOV and Why It Matters
An LOV powers dropdowns, radio groups, checkboxes, and popup searches. Defined once in Shared Components — reused on any page. If a new department is added, update the LOV in one place and every dropdown across all pages updates automatically.
LOV TypeValues Come FromExample
Static LOVYou type the options manuallyStatus: ACTIVE, INACTIVE, PENDING
Dynamic LOVSQL query — values from a DB tableSELECT dept_name, dept_id FROM departments
Popup LOVSearch popup for large datasetsSearch employee by name from 10,000 records
Create a Dynamic LOV
  1. Shared Components → List of Values → Create.
  2. Name: DEPARTMENTS_LOV. Type: Dynamic.
  3. SQL: SELECT department_name d, department_id r FROM departments ORDER BY 1
  4. Save. On any Select List item, set LOV to DEPARTMENTS_LOV.
  5. Dropdown auto-populates from DB every time the page loads.

Page Rendering and Processing — Full Detail
Core Concept

📥 Rendering — Page LOADS

  • Before Header computations run
  • Data fetched from DB
  • Items populated with values
  • Page HTML built and sent to browser
Example
Employee opens Edit form → APEX fetches their record → fields fill in automatically

📤 Processing — Page SUBMITS

  • Validations run first
  • Fail → error shown, processing stops
  • Pass → Processes (PL/SQL) execute
  • Branch → navigate to next page
Example

Part L · APEX URL Structure & Substitution Strings
🔗
APEX URL Structure & Session State Protection
Navigation
URL Format
f?p=AppID:PageNo:Session:::ItemName:Value

-- Example: Open Page 2, pass EMP_ID = 5
f?p=100:2::::P2_EMP_ID:5

100        → Application ID
2          → Page Number to open
           → Session (APEX fills automatically)
P2_EMP_ID  → Item to receive the value
5          → The value being passed
⚠ Session State Protection
Problem: User changes P2_EMP_ID:5 to P2_EMP_ID:1 in the URL and sees another employee's record.

Solution: Enable Session State Protection on the item. APEX adds a checksum. Any URL modification invalidates the checksum — APEX blocks the request and shows an error page.

Part M · Debug Mode · Error Handling · Export/Import
🐛
Debug Mode — Find What Went Wrong
Developer Tool
What is Debug Mode
Debug mode shows every step APEX took when processing a page — every SQL statement, every process, every branch decision — with execution time in milliseconds. It is your primary diagnostic tool when something does not work as expected.
How to Use
Run App
→
Click Debug (top-right)
→
Page Reloads in Debug
→
View Execution Steps
→
Find Error / Slow SQL

Or add &p_debug=YES to the URL directly. Debug shows: Authentication → Authorization → Computations → SQL → Processes → Branches — each with time taken.
Real Use Case
Your report shows no data. Debug reveals the SQL ran as WHERE dept = NULL. The item P2_DEPT was not submitted before the report ran. Fix: add P2_DEPT to Items to Submit list. Debug found the issue in 30 seconds — without debug it could take hours.

Error Handling in APEX
Reliability
Error TypeCauseHow APEX Handles It
Validation ErrorRequired field empty, wrong format, rule violatedInline error next to item. Processing stops. No data saved.
PL/SQL ExceptionUnhandled DB error (duplicate key, null constraint)Page-level error. Add EXCEPTION block to handle gracefully.
Friendly Error MessagesDefault APEX errors show ORA-XXXXX codesUse Error Handling Function in Shared Components to show user-friendly messages.
PL/SQL Exception Example
BEGIN
  INSERT INTO employees (emp_id, emp_name)
  VALUES (:P3_EMP_ID, :P3_EMP_NAME);
  COMMIT;
EXCEPTION
  WHEN DUP_VAL_ON_INDEX THEN
    apex_error.add_error(
      p_message          => 'This Employee ID already exists. Use a different ID.',
      p_display_location => apex_error.c_inline_in_notification
    );
END;

Application Export & Import — Dev to Production
Industry Practice
Why This Is Important
In every real company, developers build in a Development environment. After testing, the app is exported as a SQL file and imported into Production. This is how every enterprise APEX deployment works.
Export / Import Flow
Development Workspace
→
App Builder → Export
→
.sql File Downloaded
→
Production Workspace → Import
→
App Live in Production
  1. App Builder → select your app → click Export.
  2. Choose: Export Application Definition → click Export.
  3. A .sql file downloads — your entire application in one file.
  4. In Production workspace → App Builder → Import → upload the .sql file.
  5. Follow the install wizard. App is now live in Production.
What the Export Contains
All pages, regions, items, processes, validations, shared components, and settings. Does not contain table data — that lives in the schema separately and must be migrated independently.

Day 31 Summary

What You Learned Today

  • Oracle APEX is a low-code web platform running inside Oracle Database via ORDS — build enterprise apps using only SQL, no separate web server needed.
  • Database holds everything · Schema owns the tables · Workspace is your APEX dev environment · Application is what you build inside it.
  • Every APEX request goes through 10 stages: Authentication → Authorization → Rendering → User Interaction → Submit → Validation → Processing → Branch → Response.

  • Processing order: Computations → Validations → Processes → Branches. LOVs, Shared Components, Debug Mode, and Export/Import are daily tools in every real APEX project.
