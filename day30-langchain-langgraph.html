04 — Pipelines

LCEL — The Pipe Operator

Analogy — Swiggy Order Pipeline

You order → Restaurant confirms → Kitchen prepares → Delivery picks up → Arrives. Each step's output feeds the next automatically. You don't manage each step separately. Swiggy manages the chain.

LangChain chain = Swiggy for AI calls.

ℹ️

The | pipe symbol means "output of left flows into input of right." Like water through pipes: Input → Template → LLM → Parser → Output.

Python — LCEL New Way (Preferred)
# NEW WAY — LCEL pipe operator (preferred)
chain = template | llm | parser
result = chain.invoke({"job_title": "Java Developer"})

# OLD WAY (still works, but not recommended for new code)
# chain = LLMChain(llm=llm, prompt=template)
# result = chain.run("Java Developer")

# ── SEQUENTIAL CHAIN — two LLM calls connected ──
from langchain.chains import LLMChain, SimpleSequentialChain

extract_chain = LLMChain(llm=llm, prompt=extract_template)
jd_chain      = LLMChain(llm=llm, prompt=jd_template)

hiring_pipeline = SimpleSequentialChain(
    chains=[extract_chain, jd_chain],
    verbose=True   # shows each step output
)

result = hiring_pipeline.invoke(
    "We need a Senior Java Developer for IT team"
)
print(result["output"])
# OBSERVE: Step 1 output prints, Step 2 uses it automatically

Python — StructuredOutputParser
from langchain.output_parsers import StructuredOutputParser, ResponseSchema

schemas = [
    ResponseSchema(name="issue_type",
        description="LEAVE / SALARY / POLICY / OTHER"),
    ResponseSchema(name="urgency",
        description="HIGH / MEDIUM / LOW"),
    ResponseSchema(name="summary",
        description="One sentence summary of the complaint"),
    ResponseSchema(name="action",
        description="Recommended HR action")
]

parser = StructuredOutputParser.from_response_schemas(schemas)

template = PromptTemplate(
    template="Analyze complaint: {complaint}\n\n{format_instructions}",
    input_variables=["complaint"],
    partial_variables={"format_instructions": parser.get_format_instructions()}
)

chain  = template | llm | parser  # LCEL pipe!
result = chain.invoke({"complaint": "Salary not credited for 2 months. Urgent."})

print(result["issue_type"])  # "SALARY"
print(result["urgency"])    # "HIGH"
print(result["action"])     # "Escalate to Finance immediately"

Tools — AI That Acts

Analogy — HR Executive with Full System Access

Without tools: Brilliant expert who can only TALK. Knows all policies. Cannot check your actual leave balance. Cannot send emails. Cannot look up real data.

With tools: Same expert with database access, email, HR portal. Now she can DO things, not just recite policy. One instruction → three real actions.

Python — Defining Tools
from langchain.tools import Tool

# Regular Python function
def check_leave_balance(employee_id: str) -> str:
    try:
        balance = db.query(f"SELECT leaves FROM employees WHERE id={employee_id}")
        return f"Employee {employee_id} has {balance} days remaining."
    except Exception as e:
        return f"Error: {str(e)}"  # NEVER raise inside a tool

# Wrap as LangChain Tool — description is CRITICAL
# AI reads description to decide WHEN to call this tool
leave_tool = Tool(
    name="CheckLeaveBalance",
    description="""Check an employee's leave balance.
    Input: employee_id as string.
    Use this when asked about leave days or balance.
    DO NOT use for policy questions.""",
    func=check_leave_balance
)
⚠️

Golden rule: Always wrap tool functions in try/except and return an error string. Never raise an exception inside a tool — it will crash the agent.

✅

Real companies: Razorpay — AI calls payment gateway. Swiggy — AI calls restaurant availability. MakeMyTrip — AI calls flight booking API. Paytm — AI calls transaction history. Tools give AI the ability to ACT on real systems.

↓

↓

agent = initialize_agent(

10 — Workflow Graphs

LangGraph — Controlled AI Workflows

Analogy — HR Flowchart on the Office Wall

Every HR office has a physical flowchart: "Receive Request → Check Budget → YES: Create JD | NO: Notify CFO → Post Job → Screen Resumes..."

LangGraph builds that exact flowchart in Python — with AI doing the actual work at each box.

Three Core Concepts
STATE

Shared dictionary passed between ALL nodes. Like a job file folder travelling desk to desk. Every node reads it and adds to it. Never loses previous data.

NODES

Individual Python functions. Each is one step. Receives state, does work (LLM call or tool), returns updates to state.

EDGES

Connections between nodes. Normal: always go to next. Conditional: call a function at runtime to decide which path.

Python — Complete Hiring Workflow Graph
from typing import TypedDict, Optional
from langgraph.graph import StateGraph, END

# 1. STATE — shared file folder for the entire workflow
class HiringState(TypedDict):
    job_request:     str
    job_title:       Optional[str]
    department:      Optional[str]
    budget_approved: Optional[bool]
    jd_content:      Optional[str]

# 2. NODES — each step in the workflow
def extract_requirements(state) -> dict:
    response = llm.invoke(f"Extract job title from: {state['job_request']}")
    return {"job_title": "Senior Java Developer", "department": "IT"}

def check_budget(state) -> dict:
    budget = db.query(f"SELECT budget FROM depts WHERE name='{state['department']}'")
    return {"budget_approved": budget > 0}

def generate_jd(state) -> dict:
    jd = llm.invoke(f"Write JD for {state['job_title']} at LTIMindtree")
    return {"jd_content": jd.content}

def notify_no_budget(state) -> dict:
    send_email("cfo@ltimindtree.com", f"Budget needed for {state['job_title']}")
    return {}

# 3. DECISION FUNCTION — which path to take?
def route_budget(state) -> str:
    return "generate_jd" if state["budget_approved"] else "notify_no_budget"

# 4. BUILD THE GRAPH
graph = StateGraph(HiringState)
graph.add_node("extract",           extract_requirements)
graph.add_node("check_budget",      check_budget)
graph.add_node("generate_jd",       generate_jd)
graph.add_node("notify_no_budget",  notify_no_budget)

graph.set_entry_point("extract")
graph.add_edge("extract",           "check_budget")
graph.add_edge("generate_jd",       END)
graph.add_edge("notify_no_budget",  END)
graph.add_conditional_edges("check_budget", route_budget,
    {"generate_jd": "generate_jd", "notify_no_budget": "notify_no_budget"})

workflow = graph.compile()
result   = workflow.invoke({"job_request": "Hire Senior Java Developer for IT"})
print(result["jd_content"])

Python — Always Wrap Calls
import time

try:
    result = chain.invoke({"input": user_question})
    print(result)
except Exception as e:
    if "401" in str(e):
        print("Authentication failed — check API key")
    elif "429" in str(e):
        time.sleep(60)
        result = chain.invoke({"input": user_question})
    else:
        print(f"Error: {str(e)}")

# Inside tools — NEVER raise, always return string
def my_tool(input_str: str) -> str:

        return do_the_work(input_str)
    except Exception as e:
        return f"Tool error: {str(e)}"  # Agent reads this and handles gracefully

17 — Assessment Prep

Interview Questions

What is LangChain? Where does it fit in the AI stack?
LangChain is a Python framework for building LLM-powered applications. It sits between your app and the AI model: App → LangChain → OCI SDK → OCI Gen AI. It is NOT an AI model — it organises how your application uses AI.
Why use LangChain instead of direct OCI REST API?
Direct API = manual JSON, authentication, parsing for every call. 500+ lines for complex workflows. Breaks when model changes. LangChain = 50 lines, model-agnostic, built-in memory/parsers/tools/chains. Switch AI provider with one line.
What is LCEL and why is the pipe | operator used?
LangChain Expression Language. The pipe connects components so output of left becomes input of right automatically. prompt | llm | parser creates a pipeline. All components support invoke/batch/stream consistently. Replaces old chain.run().
Difference between Chain, Agent, and LangGraph?
Chain = fixed sequence A→B→C, no decisions. Agent = AI decides each step dynamically. LangGraph = developer defines all nodes, edges, and conditional paths, AI follows the controlled flowchart. Chain for pipelines, Agent for open Q&A, LangGraph for production workflows.
What is State in LangGraph?
A shared TypedDict passed between all nodes. Like a job file folder travelling desk to desk. Every node reads it and returns updates. Holds all information accumulated as the workflow runs. Never loses previous nodes' results.
What is an Output Parser and why use it?
Converts AI free text to structured Python dict or typed object. You define schema (fields and descriptions). Parser tells LLM what JSON to return and validates it. Makes AI output reliably machine-readable — no regex, no manual parsing.
What are the three Runnable methods?
invoke() = single call, complete response (most common). batch() = multiple parallel calls, faster for bulk operations like screening 50 resumes. stream() = progressive word-by-word output for UI typing effect.
What is model agnostic and why does it matter?
Same LangChain code works with any AI model. Change OCI Cohere to OpenAI to Gemini — one line change. Chains, memory, tools, parsers all stay unchanged. Avoids vendor lock-in. Future-proofs your codebase.
What is Memory and what are the types?
Stores conversation history so AI remembers context. BufferMemory = stores everything (short conversations under 10 turns). WindowMemory = last N messages (medium, controls cost). SummaryMemory = summarises old messages (long conversations). Without memory every question is a fresh start.
When to use Semantic Kernel instead of LangChain?
When team codes in C# or Java, when company runs on Microsoft Azure, when Microsoft enterprise support contract is needed, or when building on Microsoft 365. For Python + OCI/AWS/GCP environments — LangChain is the better choice.
Why is Tool description so important for agents?
The agent reads the description to decide WHEN to call the tool. Poor description = wrong tool chosen = wrong answer. Good description includes: what it does, when to use it, what NOT to use it for, and exact input format with examples.
What is RAG and why is it better than preambleOverride?
RAG = Retrieval Augmented Generation. AI answers from YOUR actual documents. preambleOverride = you manually type policies — can be wrong, incomplete, outdated. RAG reads the actual PDF — cannot hallucinate, handles 200 pages, always current. Full implementation in Day 37.
Temperature 0.3 vs 0.7 — when to use each?
0.3 = factual and consistent (HR policy Q&A, complaint scoring — same answer every time). 0.7 = creative and varied (JD writing, email drafting — engaging, different phrasing each call). Lower = deterministic. Higher = creative.
What is the ReAct pattern?
Reasoning + Acting. Agent loop: Thought (what do I need?) → Action (which tool?) → Observation (what did tool return?) → repeat until question fully answered. Used in ZERO_SHOT_REACT_DESCRIPTION agent type.
Input tokens vs Output tokens vs max_tokens?
Input tokens = what you SEND to AI (prompt + question). Output tokens = what AI GENERATES back (answer). Total cost = input + output. max_tokens = limit you set on output length. Set based on use case: 200 short, 500 chatbot, 1000 JD, 2000 report.
