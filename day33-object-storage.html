Root Cause

Why Not Store PDFs Locally?

Students always ask this. The answer has five clear reasons.

❌ Problem 1 — Single Machine

RAG pipeline runs on an OCI server. Arjun's laptop is in Hyderabad. The OCI server in the cloud cannot access a file on Arjun's personal laptop. Files trapped on one machine.

❌ Problem 2 — Limited Disk Space

Arjun's laptop has 512GB. LTIMindtree has 10 years of policy history — hundreds of GBs of documents, email archives, and audit logs. Laptop fills up. Then what?

❌ Problem 3 — Single Point of Failure

Arjun's laptop hard drive fails. All policy documents gone. RAG system stops working. No backup. Employees cannot get policy answers. Compliance risk.

❌ Problem 4 — No Collaboration

HR team in Delhi needs to upload a policy. They cannot access Arjun's laptop in Hyderabad. Every update requires Arjun's involvement. Developer bottleneck for every content change.

Definition

What is OCI Object Storage?

For Common Man

Think of Google Drive or Dropbox — you upload files, they live in the cloud, you can access them from anywhere. OCI Object Storage is Oracle's enterprise version, but designed for applications reading and writing files programmatically, not humans browsing folders.

For Business Person

Unlimited cloud file storage. Store any file — PDF, image, video, JSON, CSV, log. Access from anywhere in the world. Pay only for what you use. 11 nines durability means files virtually never get corrupted or lost. HR uploads a policy; RAG reads it instantly.

For Technical Person

A flat key-value store for binary data. Key = object name (acts like a file path). Value = actual file content in bytes. No real filesystem hierarchy. Accessed via REST API, OCI SDK, CLI, or Console. Scales to exabytes with no performance degradation.

Formal Definition

OCI Object Storage is a regional, highly available, highly durable cloud storage service that stores unstructured data as discrete units called objects within containers called buckets, accessible via standard REST APIs and managed through OCI Identity and Access Management.

Analogy — Post Office vs Filing Cabinet

Filing Cabinet (local storage): Physical folders in a physical drawer. You must be in the office to access it. Limited space. Gets full. One person accesses at a time. Only you can use it.

Post Office (Object Storage): You send a package with an address label. The post office stores it indefinitely. Anyone with the correct address (and permission) can pick it up from any branch in the world. Unlimited packages. Multiple people access simultaneously.

Object name = address label. Object content = the package. Bucket = the post office branch. Namespace = your company's account with the postal system.

Policy PDF → Object Storage

HR uploads PDF. RAG reads via SDK. Any server, anywhere, can access it. API-based. No mounting needed. This is OCI Object Storage.

OCI Compute instance needs a hard drive for its operating system. Block Storage is mounted as disk. Only that one VM uses it. Like a laptop's internal hard drive.

Common Misconception

There Are No Real Folders in Object Storage

⚠️

Students always get confused by this. The OCI Console shows what looks like folders. They are NOT real folders. Object Storage is a flat key-value store. The "/" in the object name is just part of the name string — not an actual directory separator.

What You See in Console
📁 hr_policy/
  📄 annual_leave.pdf
  📄 sick_leave.pdf
📁 it_policy/
  📄 laptop_policy.pdf

Looks like folders with files inside. But this is just the Console's visual presentation.

What Is Actually Stored
Key: "hr_policy/annual_leave.pdf"
Key: "hr_policy/sick_leave.pdf"
Key: "it_policy/laptop_policy.pdf"

Three flat objects. The "/" is part of the key name string. No actual directory structure exists underneath.

Analogy — Library Books with Labels

Imagine a library with no shelves. All books are on the floor. Each book has a label: "Fiction/Mystery/Book1", "Fiction/Romance/Book2", "NonFiction/Science/Book3".

There are no real shelves. The label contains "/" but that doesn't create actual sections. The librarian just groups books by label prefix when you ask. Object Storage works exactly this way.

Practical consequence: In Python SDK, filtering by "folder" means filtering by prefix: client.list_objects(namespace, bucket, prefix="hr_policy/") — this returns all objects whose key starts with "hr_policy/".

Object Management

Object Versioning & Lifecycle

Object Versioning

What: Keep all versions of every object when enabled. Upload new file with same name → new version created, old version preserved with version ID.

Why: HR updates leave policy in January. Old 2025 version must be kept for compliance audit. If new version has errors, restore old version instantly.

oci os bucket update --name ltimindtree-policies --versioning Enabled

Lifecycle Rules

What: Automatic rules that move or delete objects after a set time period.

LTIMindtree example:

  • Policy PDFs: Standard tier for 2 years
  • After 2 years → auto-move to Archive (10x cheaper)
  • After 10 years → auto-delete

Set once. Runs automatically forever. No manual work.

How SDK Works

SDK vs REST — The Relationship

🔗

Students always ask: "What does the SDK use internally?" The Python SDK is a convenient wrapper around the REST API. Every SDK call becomes an HTTP request behind the scenes. The SDK handles RSA authentication signing, request formatting, response parsing, and retry logic so you don't have to.

Without SDK (Raw REST)

You write RSA signature manually. Format HTTP headers manually. Parse JSON response manually. Handle retries manually. Complex. Error-prone. 50+ lines for one file download.

With SDK (One Line)

SDK handles everything else. Auth signing. Headers. Parsing. Retries. Same result. Much simpler.

Security

IAM Policies — Who Can Access What

🔒

Without IAM: Anyone who knows the URL can access your files. Policy PDFs exposed to the internet. Compliance disaster. With IAM: Only authorised users and services can access. Granular control: who can read, write, or delete. Full audit log of every access.

External Sharing

Pre-Authenticated Requests (PAR) & Signed URLs

Pre-Authenticated Request (PAR)

Problem: External auditor needs to access a policy PDF. No OCI account. Bucket is private.

Solution: Generate a time-limited URL. Anyone with the URL can access the file. No OCI account needed. URL expires after the set time.

Created in Console or CLI. Stored in OCI. Admin-created sharing.

Signed URL

Use case: Agent generates a report. Creates a signed URL. Emails URL to CFO. CFO clicks URL and downloads. URL expires in 1 hour.

Difference from PAR: Generated by application code on the fly. Not stored in OCI. More suitable for programmatic, automated sharing workflows.

LAB 2 Upload and Download Using Python SDK
Why this lab: Applications and AI agents use the SDK, not the CLI. This is how RAG will read policy PDFs — via client.get_object(). Note how metadata is added on upload and read back on download.
lab2_sdk.py — Python SDK operations
import oci

# Connect using ~/.oci/config (auto-authenticated in Cloud Shell)
config    = oci.config.from_file()
client    = oci.object_storage.ObjectStorageClient(config)
namespace = client.get_namespace().data

BUCKET = "ltimindtree-policies"

print(f"Connected. Namespace: {namespace}")

# UPLOAD with metadata
# Metadata helps filter in RAG and audit in compliance
policy_content = """SECTION 1: ANNUAL LEAVE POLICY
Employees are entitled to 18 days of paid annual leave per year.
Leave must be approved by the reporting manager.
Unused leave cannot be carried forward.

SECTION 2: WFH POLICY
Employees can work from home up to 2 days per week.
WFH not allowed on Mondays and Fridays without special approval."""

client.put_object(
    namespace_name=namespace,
    bucket_name=BUCKET,
    object_name="hr_policy/sdk_policy.txt",
    put_object_body=policy_content.encode(),
    opc_meta={
        "department": "HR",
        "year":       "2026",
        "status":     "Approved"
    }
)
print("Uploaded: hr_policy/sdk_policy.txt")

# LIST objects in bucket
response = client.list_objects(namespace_name=namespace, bucket_name=BUCKET)
print("\nAll objects in bucket:")
for obj in response.data.objects:
    print(f"  {obj.name}")

# DOWNLOAD and read back
response = client.get_object(
    namespace_name=namespace,
    bucket_name=BUCKET,
    object_name="hr_policy/sdk_policy.txt"
)
content = response.data.content.decode()
print("\nDownloaded content:")
print(content)

# READ METADATA
print("\nMetadata:")
print("Department:", response.headers.get("opc-meta-department"))
print("Status:",     response.headers.get("opc-meta-status"))

LAB 3 RAG Pipeline Reading from Object Storage
Why this lab: This is the key upgrade from Day 37. Instead of policy text hardcoded in Python, we now read the actual file from OCI Object Storage. One line of code replaces the entire hardcoded string. HR team can update the PDF without touching code.
lab3_rag_storage.py — RAG reading from OCI bucket
import oci
from langchain_community.chat_models.oci_generative_ai import ChatOCIGenAI
from langchain.text_splitter import RecursiveCharacterTextSplitter
import chromadb

# OCI connections
config    = oci.config.from_file()
storage   = oci.object_storage.ObjectStorageClient(config)
namespace = storage.get_namespace().data

BUCKET = "ltimindtree-policies"

llm = ChatOCIGenAI(
    model_id="cohere.command-a-03-2025",
    service_endpoint="https://inference.generativeai.ap-hyderabad-1.oci.oraclecloud.com",
    compartment_id="ocid1.compartment.oc1..aaaaaaaawsdiuqgqqkxt7wavfbqnqvf2b5kvqmmbghisajmcoqaeqhozbiea",
    auth_type="API_KEY",
    model_kwargs={"temperature": 0.1, "max_tokens": 400}
)

# STEP 1: READ FROM OBJECT STORAGE
# This one line replaces the entire hardcoded policy_text string from Day 37!
# Now HR can update the PDF without any code change.
print("Reading policy from OCI Object Storage...")
resp        = storage.get_object(namespace_name=namespace, bucket_name=BUCKET,
                              object_name="hr_policy/sample_policy.txt")
policy_text = resp.data.content.decode()
print(f"Read {len(policy_text)} characters from bucket.\n")

# STEP 2: CHUNK AND STORE IN CHROMADB (same as Day 37)
splitter   = RecursiveCharacterTextSplitter(chunk_size=300, chunk_overlap=50)
chunks     = splitter.split_text(policy_text)
client     = chromadb.Client()
collection = client.create_collection("ltimindtree_rag")
collection.add(documents=chunks, ids=[f"chunk_{i}" for i in range(len(chunks))])
print(f"Ingested {len(chunks)} chunks into ChromaDB.\n")

# STEP 3: ANSWER QUESTIONS (same as Day 37)
def ask(question):
    print(f"Q: {question}")
    results  = collection.query(query_texts=[question], n_results=3)
    context  = "\n\n".join(results["documents"][0])
    prompt   = f"""You are LTIMindtree HR Policy Assistant.
Answer ONLY from the policy text provided. Do not make up information.

POLICY TEXT:
{context}

QUESTION: {question}
ANSWER:"""
    result = llm.invoke(prompt)
    print(f"A: {result.content}\n")

ask("How many leave days per year?")
ask("Can I WFH on Friday?")
✅ Expected Output
Reading policy from OCI Object Storage... Read 892 characters from bucket. Ingested 5 chunks into ChromaDB. Q: How many leave days per year? A: You are entitled to 18 days of paid annual leave per calendar year. Q: Can I WFH on Friday? A: WFH is NOT allowed on Fridays without special approval.
✅

What changed from Day 37: Only one thing — the source of policy_text. Instead of a hardcoded string in the Python file, it now comes from OCI Object Storage via storage.get_object(). Everything else (chunking, ChromaDB, LLM) is identical. HR can now update the policy PDF without touching Python code.

Interview Preparation

Interview Questions — Beginner to Advanced

Beginner Level

What is OCI Object Storage and when would you use it?
OCI Object Storage is a cloud service for storing any type of file (PDF, image, JSON, CSV) at unlimited scale. You use it when you need files accessible from anywhere via API, need unlimited storage capacity, require high durability (11 nines), want multiple applications or users to share the same files, or need built-in versioning and lifecycle management. Common use cases: storing AI training data, policy documents for RAG pipelines, application logs, backups, and generated reports.
What is a namespace in OCI Object Storage?
A namespace is a unique identifier for your OCI tenancy within Object Storage. Every OCI account has exactly one namespace — it is globally unique across all OCI regions. All your buckets exist under this namespace. It forms part of every Object Storage URL: /n/{namespace}/b/{bucket}/o/{object}. Find yours with: oci os ns get
What is the difference between Object Storage, Block Storage, and File Storage?
Object Storage: stores files (PDFs, images, JSON) accessed via REST API. Best for AI documents, backups, application data. Block Storage: stores virtual hard disks, mounted as a disk by a single VM. Best for OS disks, databases. File Storage: stores shared NFS folders, mounted by multiple servers simultaneously. Best for shared project code or configuration. For RAG systems: policy PDFs go in Object Storage because they need to be accessed by API from anywhere.

Intermediate Level

Do buckets have real folders? How does the "/" work?
No — Object Storage is a flat key-value store. There are no real folders. The "/" in an object name like "hr_policy/annual_leave.pdf" is just part of the key string. OCI Console displays objects with "/" as if they are in folders, but this is purely a visual presentation. In the SDK, filtering by "folder" means filtering by prefix: list_objects(prefix="hr_policy/"). This returns all objects whose key starts with "hr_policy/" regardless of any actual directory structure.
What is the difference between Standard and Archive storage tiers?
Standard tier: immediate access in milliseconds, higher storage cost. Use for actively accessed files like current policy PDFs for RAG. Archive tier: requires a 1-hour restore before access, approximately 10x cheaper than Standard. Use for old policy versions, audit logs from previous years, or backup files rarely accessed. Use lifecycle rules to automatically move objects from Standard to Archive after a specified period, reducing costs without manual intervention.
What is object metadata and why is it useful for AI systems?
Object metadata is a set of custom key-value pairs attached to each object when uploading. For example: {"department": "HR", "status": "Approved", "year": "2026"}. In AI/RAG systems: metadata enables filtering retrieval to only Approved policies, filtering by department so Finance employees only get Finance policy answers, tracking which version was active on a specific date for audit, and knowing who uploaded what and when for compliance. Metadata is read back via response headers after get_object().

Advanced Level

How does the SDK relate to the REST API internally?
The Python SDK is a wrapper around the Object Storage REST API. Every SDK method call (get_object, put_object, list_objects) generates an HTTP request internally. The SDK adds value by handling RSA request signing (authentication), formatting HTTP headers correctly, parsing JSON responses into Python objects, implementing retry logic for transient failures, and providing a Pythonic interface instead of raw HTTP calls. The end result is identical — you could build everything the SDK does with raw HTTP requests, but it would require 50+ lines for what the SDK does in one line.
What IAM is needed for an AI agent to access Object Storage?
First, create a Dynamic Group that includes your compute resource (OCI Function, Container Instance, or VM) using a matching rule like resource.type = 'fnfunc'. Then create IAM policies: "Allow dynamic-group AgentGroup to read objects in compartment X where target.bucket.name='input-bucket'" for reading input documents, and "Allow dynamic-group AgentGroup to manage objects in compartment X where target.bucket.name='output-bucket'" for writing reports. This uses Instance Principal authentication — no API keys needed in the agent code, which is the most secure production approach.
Explain the three AI integration patterns with Object Storage.
Pattern 1 — Input: Object Storage feeds AI. Policy PDFs uploaded to a bucket are read by the RAG ingestion pipeline, chunked, embedded, and stored in ChromaDB. AI answers from the document content. Pattern 2 — Output: AI writes to Object Storage. Agent generates a report, saves it to an output bucket, generates a signed URL, and emails it to stakeholders. Report is permanent and auditable. Pattern 3 — Event-driven: Upload triggers AI. When a PDF is uploaded, OCI Events fires a createobject event, which invokes an OCI Function that runs the ingestion pipeline. RAG updates automatically without human intervention. All three patterns use the same SDK operations — the difference is the direction and trigger of data flow.

Key Takeaways

Day 38 Summary

🏗️ Core Concepts
  • Object Storage = flat key-value store for binary files
  • Namespace → Bucket → Object (three levels)
  • No real folders — "/" is part of the key name
  • Metadata = key-value pairs attached to each object
  • Standard tier = instant access, higher cost
  • Archive tier = 1 hour restore, 10x cheaper
  • Versioning = keep all versions for audit
  • Lifecycle rules = auto-move and auto-delete
🔧 Technical Rules
  • SDK = wrapper around REST API
  • All buckets PRIVATE — never public for sensitive data
  • Instance Principal = no keys in production code
  • PAR = time-limited URL for external sharing
  • IAM verb: read for input buckets, manage for output
  • Dynamic Group needed for agent access
  • Multipart upload for files over 100MB
  • OCI Events → auto-trigger ingestion on upload
🏢 Business Outcomes
  • No more hardcoded policy text in Python files
  • HR uploads PDF → RAG updates in 5 minutes
  • Zero developer involvement for content updates
  • Reports saved permanently (not just terminal output)
  • All versions kept for compliance audit
  • CFO gets signed URL to download reports
  • Full audit trail: who uploaded what and when
  • Lifecycle rules save 90% on old document storage
⚠️ Common Mistakes
  • Making buckets public accidentally — security risk
  • Forgetting .oci/ credentials in Docker image
  • Using Standard tier for 10-year archive — expensive
  • Assuming Object Storage = Vector Database (they're different)
  • Confusing namespace with bucket name
  • Not enabling versioning on policy buckets
  • Using CLI path syntax with "/" expecting real folders
  • Using same embedding model for ingestion and query
