Introduction to PL/SQL

What PL/SQL is, how it differs from SQL, its advantages, and internal architecture

1.1

What is PL/SQL?

Oracle's procedural extension to standard SQL — runs entire code blocks directly on the database server

PL/SQL stands for Procedural Language / Structured Query Language. It is Oracle Corporation's proprietary extension to standard SQL that adds procedural constructs — loops, conditional statements, variables, and exception handling — to create powerful database programs.

Instead of sending individual SQL statements one at a time, a PL/SQL block is sent as a single unit, compiled once, and executed entirely on the database server — eliminating repeated network round-trips.

hello_plsql.sql
-- Enable script output display
SET SERVEROUTPUT ON;

BEGIN
    DBMS_OUTPUT.PUT_LINE('Hello, PL/SQL!');
END;
/
Output
Hello, PL/SQL!
💡 Real-world Example
  • Writing an automated script that loops through late payments and calculates interest penalties for each invoice — all inside a single PL/SQL block
1.2

SQL vs PL/SQL

Declarative set-based retrieval vs procedural row-by-row processing

FeatureSQLPL/SQL
ParadigmDeclarative — say WHAT you wantProcedural — say HOW to process it
ExecutionOne statement at a timeEntire block at once
VariablesNot supportedFully supported
ConditionalsCASE expression onlyIF / ELSIF / CASE statements
LoopsNot supportedLOOP / WHILE / FOR
Error HandlingError terminates statementFull EXCEPTION block
Best ForSimple data retrievalBusiness logic & complex processing
💡

SQL fetches an employee row. PL/SQL uses that row's data to decide whether to trigger an email or run a secondary bonus calculation.

1.3

Advantages of PL/SQL

Performance, security, modularity, and exception safety

AdvantageHow it Helps
PerformanceOne network trip instead of 100 — entire block executes on the server
SecurityUsers interact with stored procedures, never directly with base tables
ModularityCode reuse through procedures, functions, and packages
Exception SafetyRuntime errors caught and handled gracefully without crashing

💡

An e-commerce app checks inventory, creates an invoice, and updates a ledger inside one PL/SQL routine instead of making three separate slow server trips.

1.4

PL/SQL Architecture

How Oracle splits procedural code from SQL and routes each to the right engine

StepWhat HappensEngine
1Application sends the full PL/SQL blockApplication Layer
2PL/SQL Engine receives and parses the blockPL/SQL Engine
3Procedural code (IF, LOOP, :=) executes locallyPL/SQL Engine
4SQL statements are extracted and forwardedPL/SQL Engine → SQL Engine
5SQL Engine optimises and executes the querySQL Statement Executor
6Result returned to PL/SQL Engine to continuePL/SQL Engine
💡

A loop that repeats an INSERT five times is controlled by the PL/SQL engine, but each actual INSERT is pushed to the SQL engine for execution.

PL/SQL Block Structure

Every PL/SQL program is organised into structured sections: DECLARE, BEGIN, EXCEPTION, END

2.0

Block Overview

Visual structure of all four sections with mandatory vs optional labels

DECLARE Optional
Define variables, constants, cursors, and data types before execution begins
BEGIN Mandatory
Core SQL statements, business logic, calculations, and procedural instructions
EXCEPTION Optional
Intercept runtime errors — run fallback code or log bugs instead of crashing
END; Mandatory
Marks the final boundary of the logical code unit. Always followed by /
💡

Only BEGIN and END are mandatory. DECLARE and EXCEPTION are optional but almost always used in real-world code.

2.1

DECLARE Section

Set up variables, data types, and constants before any execution begins

The optional entry segment of a block. Everything defined here is available throughout the BEGIN and EXCEPTION sections of the same block. Variables declared in a nested block are local to that nested block only.

declare_section.sql
DECLARE
    v_test VARCHAR2(20) := 'Testing Declare';
BEGIN
    DBMS_OUTPUT.PUT_LINE(v_test);
END;
/
Output
Testing Declare
2.2

BEGIN Section

The mandatory execution section containing all actual logic and SQL statements

The mandatory execution segment that contains all actual query statements, business logic, calculations, and instructions. At least one executable statement must exist between BEGIN and END.

begin_section.sql
BEGIN
    DBMS_OUTPUT.PUT_LINE(10 + 20);
END;
/
Output
30
2.3

EXCEPTION Section

The safety net — intercepts runtime errors so the program doesn't crash

The optional safety segment dedicated to intercepting runtime errors. When an error occurs in BEGIN, execution jumps immediately to the matching EXCEPTION handler. Without this section, any runtime error would terminate the entire block abruptly.

exception_section.sql
DECLARE
    v_num NUMBER := 10;
BEGIN
    v_num := v_num / 0;  -- Fails immediately
EXCEPTION
    WHEN ZERO_DIVIDE THEN
        DBMS_OUTPUT.PUT_LINE('Error: Cannot divide by zero.');
END;
/
Output
Error: Cannot divide by zero.
2.4

END Statement

Mandatory keyword that marks the final boundary of the code block

The mandatory closing keyword that indicates the end of the current logical code unit. In SQL*Plus and SQL Developer, the forward slash / on the next line submits the block for execution.

end_statement.sql
BEGIN
    DBMS_OUTPUT.PUT_LINE('Done');
END;  -- Mandatory end marker
/     -- Submits block for execution
Output
Done

Variables

Declaration, initialization, scope, constants, %TYPE and %ROWTYPE anchors

3.1

Variable Declaration

Creating a named storage container with a specific data type in DECLARE

declaration.sql
DECLARE
    v_count  NUMBER;           -- declared, starts as NULL
    v_name   VARCHAR2(50);     -- text variable, max 50 chars
    v_active BOOLEAN;          -- logical flag
BEGIN
    v_count := 5;
    DBMS_OUTPUT.PUT_LINE(v_count);
END;
/
Output
5
💡

Syntax: variable_name datatype [NOT NULL] [:= value]; — All variables start as NULL unless explicitly initialized.

3.2

Initialization

Assigning a starting value at declaration using := or the DEFAULT keyword

initialization.sql
DECLARE
    v_points   NUMBER      := 100;              -- using :=
    v_status   VARCHAR2(10) DEFAULT 'ACTIVE';  -- using DEFAULT
    v_required NUMBER      NOT NULL := 0;      -- NOT NULL must have value
BEGIN
    DBMS_OUTPUT.PUT_LINE(v_points || ' - ' || v_status);
END;
/
Output
100 - ACTIVE
3.3

Scope of Variables

Outer block variables are visible inside; inner block variables are private

scope.sql
DECLARE
    v_global VARCHAR2(20) := 'Global';
BEGIN
    DECLARE
        v_local VARCHAR2(20) := 'Local';  -- only visible here
    BEGIN
        DBMS_OUTPUT.PUT_LINE(v_global || ' and ' || v_local);
    END;
    -- v_local is invisible here — would cause compile error
    DBMS_OUTPUT.PUT_LINE(v_global);
END;
/
Output
Global and Local
Global
3.4

Constants

Values locked at declaration — cannot be changed anywhere in the code

constants.sql
DECLARE
    co_tax_rate CONSTANT NUMBER      := 0.05;
    co_company  CONSTANT VARCHAR2(50) := 'Oracle Corp';
BEGIN
    -- co_tax_rate := 0.08;  -- ERROR: cannot modify a constant
    DBMS_OUTPUT.PUT_LINE('Tax rate: ' || co_tax_rate);
    DBMS_OUTPUT.PUT_LINE('Company : ' || co_company);
END;
/
Output
Tax rate: .05
Company : Oracle Corp
💡

Naming convention: prefix constants with co_ or c_ to distinguish them from regular variables at a glance.

3.5

%TYPE Anchor

Automatically match a variable's type to a specific table column — adapts if the column changes

Instead of hardcoding VARCHAR2(50), anchor the variable directly to the column it will receive data from. If the column type changes, the variable adapts automatically — no code changes needed.

type_anchor.sql
DECLARE
    -- Without %TYPE (fragile — breaks if column changes)
    v_name_bad  VARCHAR2(50);

    -- With %TYPE (robust — auto-matches column definition)
    v_emp_name  employees.first_name%TYPE;
    v_salary    employees.salary%TYPE;
    v_dept_id   employees.department_id%TYPE;

BEGIN
    SELECT first_name, salary, department_id
    INTO   v_emp_name, v_salary, v_dept_id
    FROM   employees
    WHERE  employee_id = 101;

    DBMS_OUTPUT.PUT_LINE(v_emp_name || ' | $' || v_salary);
END;
/
✅

Best practice: Always use %TYPE for variables that will hold column data. It prevents truncation errors and makes code maintainable across schema changes.

3.6

%ROWTYPE Anchor

One variable that holds all columns of an entire table row

%ROWTYPE declares a record variable that matches the structure of an entire table row — one variable instead of fifteen. Access individual fields using dot notation.

rowtype_anchor.sql
DECLARE
    v_emp employees%ROWTYPE;  -- one variable = all columns
BEGIN
    SELECT * INTO v_emp
    FROM   employees
    WHERE  employee_id = 101;

    -- Access fields with dot notation
    DBMS_OUTPUT.PUT_LINE('Name : ' || v_emp.first_name || ' ' || v_emp.last_name);
    DBMS_OUTPUT.PUT_LINE('Salary: $' || v_emp.salary);
    DBMS_OUTPUT.PUT_LINE('Dept  : ' || v_emp.department_id);
END;
/
💡

%TYPE vs %ROWTYPE: Use %TYPE for individual column variables. Use %ROWTYPE when you need all (or most) columns from a row — it simplifies SELECT * INTO statements significantly.

VARCHAR2

Variable-length text — occupies only as much space as the actual content

varchar2_type.sql
DECLARE
    v_name  VARCHAR2(30) := 'Alice Smith';
    v_email VARCHAR2(100);
BEGIN
    v_email := LOWER(v_name) || '@company.com';

    DBMS_OUTPUT.PUT_LINE(v_email);
END;
/

4.3

CHAR

Fixed-length text — Oracle pads shorter values with spaces to fill the defined size

char_type.sql
DECLARE
    v_code CHAR(5) := 'IN';  -- stored as 'IN   ' (3 trailing spaces)
BEGIN
    DBMS_OUTPUT.PUT_LINE(v_code || 'X');  -- 'IN   X'
END;
/
Output
IN X
⚠️

Use CHAR only for truly fixed-size codes (country codes, status flags). For general text, VARCHAR2 is almost always the right choice.

4.4

DATE

Stores century, year, month, day, hour, minute, and second — no timezone

date_type.sql
DECLARE
    v_today    DATE := SYSDATE;
    v_deadline DATE := DATE '2026-12-31';
    v_diff     NUMBER;
BEGIN
    v_diff := v_deadline - v_today;   -- difference in days
    DBMS_OUTPUT.PUT_LINE('Today   : ' || TO_CHAR(v_today, 'DD-MON-YYYY'));
    DBMS_OUTPUT.PUT_LINE('Deadline: ' || TO_CHAR(v_deadline, 'DD-MON-YYYY'));
    DBMS_OUTPUT.PUT_LINE('Days left: ' || FLOOR(v_diff));
END;
/
Output
Today : 14-JUN-2026
Deadline: 31-DEC-2026
Days left: 200
4.5

BOOLEAN

Stores TRUE, FALSE, or NULL — exists only in PL/SQL, not in SQL table columns

boolean_type.sql
DECLARE
    v_is_active BOOLEAN := TRUE;
BEGIN
    -- Cannot print BOOLEAN directly — must use IF
    IF v_is_active THEN
        DBMS_OUTPUT.PUT_LINE('User is active');
    ELSE
        DBMS_OUTPUT.PUT_LINE('User is inactive');
    END IF;
END;
/
Output
User is active
⚠️

BOOLEAN cannot be passed to DBMS_OUTPUT.PUT_LINE directly. Always wrap it in an IF statement to convert to a printable string.

5.5

NULL Literal

NULL means unknown or missing — it is NOT zero, NOT empty string, NOT false

null_literal.sql
DECLARE
    v_bonus NUMBER;             -- starts as NULL by default
    v_total NUMBER;
BEGIN
    v_total := 1000 + v_bonus;  -- NULL arithmetic = NULL!
    DBMS_OUTPUT.PUT_LINE('Total: ' || NVL(v_total, 0));

    -- Correct way to check NULL
    IF v_bonus IS NULL THEN
        DBMS_OUTPUT.PUT_LINE('No bonus assigned');
    END IF;
END;
/
Output
Total: 0
No bonus assigned
🚫

NULL traps: NULL = NULL is never TRUE. 10 + NULL = NULL. v_bonus = NULL never matches. Always use IS NULL and NVL().

Two-path branching — one block when TRUE, another when FALSE or NULL

DECLARE

6.3

IF-ELSIF

Check multiple conditions in sequence — executes first TRUE branch, skips the rest

if_elsif.sql
DECLARE
    v_marks NUMBER := 75;
BEGIN
    IF     v_marks >= 90 THEN DBMS_OUTPUT.PUT_LINE('Grade A');
    ELSIF  v_marks >= 70 THEN DBMS_OUTPUT.PUT_LINE('Grade B');  -- stops here
    ELSIF  v_marks >= 50 THEN DBMS_OUTPUT.PUT_LINE('Grade C');
    ELSE                      DBMS_OUTPUT.PUT_LINE('Fail');
    END IF;
END;
/
Output
Grade B
6.4

CASE — Simple (Selector)

Evaluates a specific variable against exact values — cleaner than long IF-ELSIF chains

case_simple.sql
DECLARE
    v_tier CHAR(1) := 'G';
BEGIN
    CASE v_tier
        WHEN 'G' THEN DBMS_OUTPUT.PUT_LINE('Gold Tier');
        WHEN 'S' THEN DBMS_OUTPUT.PUT_LINE('Silver Tier');
        WHEN 'B' THEN DBMS_OUTPUT.PUT_LINE('Bronze Tier');
        ELSE          DBMS_OUTPUT.PUT_LINE('Standard Tier');
    END CASE;
END;
/
Output
Gold Tier
6.5

CASE — Searched (Condition-Based)

No selector variable — each WHEN has its own full condition expression

Unlike simple CASE which matches one variable to values, searched CASE evaluates a completely independent condition in each WHEN clause — allowing ranges, multi-column conditions, and complex expressions.

case_searched.sql
DECLARE
    v_salary NUMBER := 75000;
    v_grade  VARCHAR2(10);
BEGIN
    CASE
        WHEN v_salary >= 100000 THEN v_grade := 'Executive';
        WHEN v_salary >=  70000 THEN v_grade := 'Senior';    -- matches
        WHEN v_salary >=  40000 THEN v_grade := 'Mid-level';
        ELSE                         v_grade := 'Junior';
    END CASE;
    DBMS_OUTPUT.PUT_LINE('Grade: ' || v_grade);
END;
/
Output
Grade: Senior
💡

Simple CASE matches one selector to fixed values (like a switch statement). Searched CASE evaluates independent conditions in each WHEN — use it for ranges and complex logic.

6.6

Nested IF

An IF inside another IF — secondary conditions checked only after primary conditions pass

nested_if.sql
DECLARE
    v_is_open BOOLEAN := TRUE;
    v_stock   NUMBER  := 2;
BEGIN
    IF v_is_open THEN              -- Gate 1: is store open?
        IF v_stock > 0 THEN         -- Gate 2: is item in stock?
            DBMS_OUTPUT.PUT_LINE('Buy Item');
        ELSE
            DBMS_OUTPUT.PUT_LINE('Out of stock');
        END IF;
    ELSE
        DBMS_OUTPUT.PUT_LINE('Store is closed');
    END IF;
END;
/
Output
Buy Item
⚠️

Keep nesting to a maximum of 3–4 levels. Deeper nesting becomes hard to read and debug — consider combining conditions with AND/OR or using a CASE statement instead.

EXIT WHEN

Concise single-line exit — checks condition and exits if TRUE

exit_when.sql
DECLARE
    v_cnt NUMBER := 1;

        DBMS_OUTPUT.PUT_LINE(v_cnt);

7.8

Nested Loops with Labels

Loop labels let you EXIT or CONTINUE a specific outer loop from inside an inner loop

nested_loops.sql
BEGIN
    <<outer_loop>>
    FOR i IN 1..3 LOOP
        <<inner_loop>>
        FOR j IN 1..3 LOOP
            EXIT outer_loop WHEN i = 2 AND j = 2;  -- exits BOTH loops
            DBMS_OUTPUT.PUT_LINE('i=' || i || ' j=' || j);
        END LOOP inner_loop;
    END LOOP outer_loop;
    DBMS_OUTPUT.PUT_LINE('After both loops');
END;
/
Output
i=1 j=1
i=1 j=2
i=1 j=3
i=2 j=1
After both loops
💡

Without labels, EXIT only exits the innermost loop. Labels allow you to target any enclosing loop directly.

Exception Handling

Predefined, user-defined, RAISE, WHEN OTHERS, SQLCODE/SQLERRM, PRAGMA, RAISE_APPLICATION_ERROR

8.1

What is an Exception?

A runtime error event that halts normal execution and transfers control to the EXCEPTION section

An exception is a runtime warning or error triggered when an operation fails during execution. When raised, Oracle immediately halts the current statement, skips the remaining BEGIN code, and transfers control to the matching handler in the EXCEPTION section.

ExceptionORA ErrorWhen Raised
NO_DATA_FOUNDORA-01403SELECT INTO returns 0 rows
TOO_MANY_ROWSORA-01422SELECT INTO returns more than 1 row
ZERO_DIVIDEORA-01476Division by zero
DUP_VAL_ON_INDEXORA-00001Insert duplicate unique value
VALUE_ERRORORA-06502Variable too small for the data
INVALID_NUMBERORA-01722String to number conversion fails
8.2

Predefined Exceptions

Built-in Oracle exceptions with ready-made names — catch them directly by name

predefined_exc.sql
DECLARE
    v_name VARCHAR2(50);
BEGIN
    SELECT emp_name INTO v_name
    FROM   employees
    WHERE  emp_id = -999;            -- ID that doesn't exist
EXCEPTION
    WHEN NO_DATA_FOUND THEN
        DBMS_OUTPUT.PUT_LINE('No record found for that ID.');
    WHEN TOO_MANY_ROWS THEN
        DBMS_OUTPUT.PUT_LINE('Multiple rows returned — use a cursor.');
END;
/
Output
No record found for that ID.
8.3

User-Defined Exceptions

Custom exceptions that enforce business rules — declared in DECLARE, raised with RAISE

user_defined_exc.sql
DECLARE
    ex_insufficient_funds EXCEPTION;   -- custom business rule
    v_balance NUMBER := 30;
BEGIN
    IF v_balance < 50 THEN
        RAISE ex_insufficient_funds;    -- manually trigger
    END IF;
EXCEPTION
    WHEN ex_insufficient_funds THEN
        DBMS_OUTPUT.PUT_LINE('Transaction Denied: Balance too low.');
END;
/
Output
Transaction Denied: Balance too low.
8.4

RAISE Statement

Explicitly triggers an exception — user-defined, predefined, or re-raises current error

raise_stmt.sql
BEGIN
    RAISE INVALID_NUMBER;   -- raise a predefined system exception
EXCEPTION
    WHEN INVALID_NUMBER THEN
        DBMS_OUTPUT.PUT_LINE('Caught an invalid number event.');
        RAISE;   -- re-raise to propagate to caller
END;
/
Output
Caught an invalid number event.

8.5

WHEN OTHERS

Catch-all handler — intercepts any error not matched by earlier WHEN clauses

when_others.sql
DECLARE
    v_num NUMBER;
BEGIN
    v_num := 'This is text, not a number';  -- VALUE_ERROR
EXCEPTION
    WHEN NO_DATA_FOUND THEN
        DBMS_OUTPUT.PUT_LINE('Found no data.');
    WHEN OTHERS THEN
        DBMS_OUTPUT.PUT_LINE('Unexpected error occurred.');
END;
/
Output
Unexpected error occurred.
🚫

Never write WHEN OTHERS THEN NULL; — this silently swallows the error. Always log SQLERRM or re-raise.

8.6

SQLCODE & SQLERRM

Built-in functions that return the error number and message of the current exception

SQLCODE returns the numeric error code (negative for most Oracle errors). SQLERRM returns the full human-readable error message. Both are only meaningful inside an EXCEPTION handler.

sqlcode_sqlerrm.sql
DECLARE
    v_num NUMBER;
BEGIN
    v_num := 10 / 0;               -- ZERO_DIVIDE error
EXCEPTION
    WHEN OTHERS THEN
        DBMS_OUTPUT.PUT_LINE('Error Code   : ' || SQLCODE);
        DBMS_OUTPUT.PUT_LINE('Error Message: ' || SQLERRM);

        -- Log to error table
        INSERT INTO error_log (err_code, err_msg, err_date)
        VALUES (SQLCODE, SQLERRM, SYSDATE);
        COMMIT;
END;
/
Output
Error Code : -1476
Error Message: ORA-01476: divisor is equal to zero
💡

Use SQLCODE and SQLERRM together in WHEN OTHERS to log unexpected errors. Insert them into an error_log table for production monitoring.

8.7

PRAGMA EXCEPTION_INIT

Associates a user-defined exception name with a specific Oracle error number

By default, user-defined exceptions have no error number. PRAGMA EXCEPTION_INIT binds a named exception to a specific ORA- error code, allowing you to catch Oracle system errors by a meaningful business name.

pragma_init.sql
DECLARE
    e_child_records  EXCEPTION;
    PRAGMA EXCEPTION_INIT(e_child_records, -2292);  -- ORA-02292

    e_dup_value      EXCEPTION;
    PRAGMA EXCEPTION_INIT(e_dup_value, -1);          -- ORA-00001

BEGIN
    DELETE FROM departments WHERE department_id = 10;
    COMMIT;
EXCEPTION
    WHEN e_child_records THEN
        DBMS_OUTPUT.PUT_LINE('Cannot delete: employee records exist in this department.');
        ROLLBACK;
    WHEN e_dup_value THEN
        DBMS_OUTPUT.PUT_LINE('Duplicate value — record already exists.');
END;
/
Output
Cannot delete: employee records exist in this department.
💡

PRAGMA must be in the DECLARE section. The error number is always negative (e.g., -2292 for ORA-02292).

8.8

RAISE_APPLICATION_ERROR

Create a custom error number (-20000 to -20999) with a descriptive message visible to calling applications

Unlike RAISE which only works within PL/SQL, RAISE_APPLICATION_ERROR surfaces the error to the calling application — Java, Python, APEX — with a custom error code and message, making it the standard for business rule enforcement in stored procedures.

raise_app_error.sql
DECLARE
    v_salary    NUMBER := 260000;
    v_hire_date DATE   := DATE '2028-01-01';
    c_max_sal   CONSTANT NUMBER := 250000;
BEGIN
    -- Validate hire date
    IF v_hire_date > SYSDATE THEN
        RAISE_APPLICATION_ERROR(-20001,
            'Hire date cannot be in the future: '
            || TO_CHAR(v_hire_date, 'DD-MON-YYYY'));
    END IF;

    -- Validate salary cap
    IF v_salary > c_max_sal THEN
        RAISE_APPLICATION_ERROR(-20002,
            'Salary $' || v_salary || ' exceeds max $' || c_max_sal);
    END IF;
EXCEPTION
    WHEN OTHERS THEN
        DBMS_OUTPUT.PUT_LINE(SQLERRM);
END;
/
Output
ORA-20001: Hire date cannot be in the future: 01-JAN-2028
📋

Numbering convention: Organise error ranges by module — e.g., -20001 to -20099 for employee validation, -20100 to -20199 for order processing, -20900 to -20999 for system errors.

SELECT INTO

Retrieve exactly one SQL row into PL/SQL variables — the bridge between SQL and PL/SQL

9.1

Basic SELECT INTO

Fetch column values from exactly one row directly into PL/SQL variables

SELECT INTO is the PL/SQL mechanism to retrieve data from the database into variables. It must return exactly one row — zero rows raises NO_DATA_FOUND, more than one raises TOO_MANY_ROWS.

select_into_basic.sql
DECLARE
    v_emp_id   employees.employee_id%TYPE := 101;
    v_name     employees.last_name%TYPE;
    v_salary   employees.salary%TYPE;
    v_dept     departments.department_name%TYPE;
BEGIN
    -- Single row fetch with JOIN
    SELECT e.last_name, e.salary, d.department_name
    INTO   v_name, v_salary, v_dept
    FROM   employees e
    JOIN   departments d ON e.department_id = d.department_id
    WHERE  e.employee_id = v_emp_id;

    DBMS_OUTPUT.PUT_LINE('Name  : ' || v_name);
    DBMS_OUTPUT.PUT_LINE('Dept  : ' || v_dept);
    DBMS_OUTPUT.PUT_LINE('Salary: $' || v_salary);
END;
/
Output
Name : King
Dept : Executive
Salary: $24000
9.2

SELECT INTO — Error Handling

NO_DATA_FOUND and TOO_MANY_ROWS must always be handled

select_into_errors.sql
DECLARE
    v_name employees.first_name%TYPE;
BEGIN
    SELECT first_name INTO v_name
    FROM   employees
    WHERE  employee_id = 99999;   -- does not exist

    DBMS_OUTPUT.PUT_LINE('Found: ' || v_name);

EXCEPTION
    WHEN NO_DATA_FOUND THEN
        DBMS_OUTPUT.PUT_LINE('Employee not found.');

    WHEN TOO_MANY_ROWS THEN
        DBMS_OUTPUT.PUT_LINE('Multiple rows returned — add WHERE condition.');

    WHEN OTHERS THEN
        DBMS_OUTPUT.PUT_LINE('Error: ' || SQLERRM);
END;
/
Output
Employee not found.
9.3

Safe SELECT INTO Patterns

Techniques that guarantee a single row is returned — avoiding both error types

Pattern 1 — Use Aggregate (Always returns 1 row)
safe_aggregate.sql
DECLARE
    v_avg_sal NUMBER;
    v_emp_cnt NUMBER;
BEGIN
    -- Aggregate ALWAYS returns 1 row (NULL if no data)
    SELECT AVG(salary), COUNT(*)
    INTO   v_avg_sal, v_emp_cnt
    FROM   employees
    WHERE  department_id = 50;

    DBMS_OUTPUT.PUT_LINE('Avg Salary: $' || NVL(v_avg_sal, 0));
    DBMS_OUTPUT.PUT_LINE('Headcount : ' || v_emp_cnt);
END;
/
Pattern 2 — SELECT INTO with %ROWTYPE
safe_rowtype.sql
DECLARE
    v_emp employees%ROWTYPE;   -- all columns in one variable
BEGIN
    SELECT * INTO v_emp
    FROM   employees
    WHERE  employee_id = 101;  -- primary key guarantees 1 row

    DBMS_OUTPUT.PUT_LINE(v_emp.first_name || ' earns $' || v_emp.salary);
EXCEPTION
    WHEN NO_DATA_FOUND THEN
        DBMS_OUTPUT.PUT_LINE('Employee 101 not found.');
END;
/
✅

Rule: Always use a PRIMARY KEY or UNIQUE column in the WHERE clause of SELECT INTO. This mathematically guarantees at most one row — eliminating TOO_MANY_ROWS.
