What is Artificial Intelligence?

Imagine you ask your phone: "What's the weather in Chennai tomorrow?" — Your phone understands your sentence, connects to a weather service, and replies in plain English. No human helped it. That ability — a machine understanding, reasoning, and responding like a human — is Artificial Intelligence.

Formally: AI is the science of making machines perform tasks that normally require human intelligence — understanding language, recognising patterns, making decisions, and solving problems.

🔷 Definition

Artificial Intelligence

Machines that simulate human cognitive functions — learning, reasoning, problem-solving, perception, and language understanding.

⚡ Why It Exists

The Problem It Solved

Manual rule-based systems broke the moment reality didn't match the rules. AI learns from data — no hardcoding needed. Works even on edge cases.

📚 Before AI

Rule-Based Systems (Expert Systems)

Engineers manually wrote IF-THEN rules. A medical diagnosis system had 10,000+ hand-coded rules. Broke down with new diseases or edge cases.

Machine Learning — Learning From Data

A traditional program for spam detection would look like: IF email contains "lottery" OR "winner" THEN mark spam. This fails the moment spammers write "l0ttery" or use images. Machine Learning instead learns from thousands of examples of spam and non-spam emails — it finds its own patterns without being told what they are.

🔷 Definition

ML is the field where algorithms learn patterns from data to make decisions or predictions — without explicit rule programming.

⚡ Why It Exists

Real-world problems have millions of edge cases. Writing rules for each is impossible. ML finds rules automatically from examples.

📚 Before ML

Expert Systems required domain experts to write every rule manually. MYCIN (medical diagnosis, 1970s) had 600+ handcrafted rules — still missed rare cases.

🗂️ Types of Machine Learning

1️⃣ Supervised Learning

Learns from labelled examples

Input → Label: 10,000 emails labelled spam/not-spam → model learns to classify new emails

  • Classification (spam? fraud? disease?)
  • Regression (what price? what demand?)
2️⃣ Unsupervised Learning

Finds hidden structure

Input → Pattern: 1M customer purchase histories → model discovers 5 customer segments automatically

Learns by trial and reward

Agent → Action → Reward: AI plays 1M chess games, wins get +1, losses get -1, learns best strategy

  • Game playing (AlphaGo)
  • Robot control, trading bots

Deep Learning — When Features Learn Themselves

Traditional ML requires a human expert to decide which features matter. To detect cancer in an X-ray, a radiologist must tell the ML model: "look for irregular edges, density variations, calcifications..." This is called feature engineering and it's time-consuming and imperfect.

Deep Learning eliminated this problem. You feed in raw pixels. The neural network figures out by itself that edges matter, then textures, then shapes, then structures. No human has to define what to look for.

🔷 What

ML using multi-layer artificial neural networks that automatically learn hierarchical features from raw data. The "deep" refers to many layers (10–1000+).

⚡ Why

Traditional ML failed at images, audio, text — too complex for human feature engineering. Deep Learning processes raw data end-to-end and outperforms humans on many tasks.

📚 Before DL

SVM, Random Forest needed manual features. For face recognition, you'd manually code edge detectors, then eye detectors... Deep Learning replaced all of this with one model trained end-to-end.

Neural Networks — How a Brain-Inspired Machine Learns

The human brain has ~86 billion neurons. Each neuron receives signals, processes them, and fires if the signal is strong enough. Artificial neural networks are a mathematical abstraction of this idea — simplified, but powerful enough to learn language, vision, and reasoning.

🔬 The Neuron — Building Block

BIOLOGICAL NEURON: ARTIFICIAL NEURON (Perceptron): Dendrites (inputs) x₁ ─── w₁ ─┐ │ x₂ ─── w₂ ─┤─── Σ(wᵢxᵢ + b) → activation() → output Cell body (sums) x₃ ─── w₃ ─┘ │ Axon (output fires Where: if threshold met) x = input values w = weights (what model learns) b = bias (adjustable threshold) activation = ReLU/Sigmoid/Tanh

🔄 Training — Backpropagation Explained Simply

STEP 1: FORWARD PASS Input → Layer1 → Layer2 → ... → Output (e.g., predicted = 0.3) STEP 2: CALCULATE LOSS (how wrong are we?) Loss = (actual - predicted)² = (1.0 - 0.3)² = 0.49 ← too high! STEP 3: BACKPROPAGATION "Which weights contributed most to this error?" Gradient flows BACKWARD through all layers → each weight gets its blame STEP 4: WEIGHT UPDATE (Gradient Descent) new_weight = old_weight - (learning_rate × gradient) = 0.5 - (0.01 × 0.49) = 0.4951 ← slightly better STEP 5: REPEAT millions of times Each iteration: weights get a tiny bit better → loss decreases → model learns
Activation Functions

Why needed?

Without activation, neural networks = one big linear equation (useless). Activations add non-linearity so networks can learn complex patterns.

ReLU: max(0,x) — most common
Sigmoid: 0–1 output (binary classification)
Softmax: probabilities that sum to 1

Loss Functions

Measuring wrongness

MSE: Regression problems (price prediction)
Cross-Entropy: Classification (spam/not spam)
KL Divergence: Distribution matching (used in LLM training!)

Optimizers

How to update weights

SGD: Basic gradient descent
Adam: Adaptive learning rates — most popular
AdamW: Adam + weight decay — used to train all modern LLMs

The Transformer Architecture (2017)

Before 2017, the best NLP models were RNNs (Recurrent Neural Networks). They read text one word at a time, left to right. The problem: by the time an RNN reached word 200 in a long document, it had almost completely forgotten word 1. This is called the vanishing gradient problem.

In 2017, Google researchers published "Attention Is All You Need" — introducing the Transformer. Instead of reading one word at a time, a Transformer looks at ALL words simultaneously and computes how much attention each word should pay to every other word. This single idea unlocked GPT, BERT, Claude, and every modern LLM.

📚 Before (RNN/LSTM)
  • Reads text word-by-word sequentially
  • Forgets distant context (long sentences)
  • Cannot be parallelised (slow to train)
  • Best: 512-token context window
⚡ Transformer Advantage
  • Reads all words at once (parallel)
  • Any word can attend to any other
  • Massively parallelisable → GPU-friendly
  • Context: 8K → 128K → 1M+ tokens
🏆 Impact
  • Foundation of ALL modern LLMs
  • BERT (2018), GPT-2 (2019), GPT-3 (2020)
  • Extends to vision (ViT), audio, video
  • The most cited architecture in AI history

🔍 Self-Attention — The Core Insight

Consider the sentence: "The bank can guarantee deposits will eventually cover future tuition costs because it was established before the recent financial crisis."

The word "it" refers to "bank." Humans understand this. RNNs often fail over long distances. Self-attention solves this by computing a score between every pair of words — "it" and "bank" get a high attention score because the context makes them related.

SELF-ATTENTION: "How much should each word attend to every other word?" Sentence: "The bank guaranteed deposits" ↑ ↑ ↑ ↑ For word "guaranteed": ┌─────────────────────────────────────────────────┐ │ Attention to "The" = 0.05 (low) │ │ Attention to "bank" = 0.71 (HIGH!) │ │ Attention to "guaranteed" = 0.15 (self) │ │ Attention to "deposits" = 0.09 (moderate) │ └─────────────────────────────────────────────────┘ HOW IT WORKS — The QKV Mechanism: Every word is projected into 3 vectors: Q (Query) = "What am I looking for?" K (Key) = "What do I represent?" V (Value) = "What information do I carry?" Attention Score = softmax(Q × Kᵀ / √dₖ) × V ────────────────────────── "How similar is my query to each key?" Scaled by √dₖ to prevent vanishing gradients softmax → probabilities that sum to 1

Types of LLMs — Encoder, Decoder, Encoder-Decoder
📥 Encoder-Only

Understand text deeply

Attention: Bidirectional (reads ALL context simultaneously)

Task: Produces rich token representations — not text generation

Used for: Classification, NER, Sentiment, Embeddings, Semantic Search

Examples: BERT, RoBERTa, DistilBERT

Think of it as: A brilliant reader who understands everything but can't write

📤 Decoder-Only

Generate text token by token

Attention: Causal / Masked (can only see previous tokens)

Task: Autoregressive text generation — next token prediction

Used for: Chatbots, Code generation, Document drafting, RAG, Reasoning

Examples: GPT-4, LLaMA, Mistral, Claude, Cohere Command R

Think of it as: A brilliant writer who generates word by word

↔️ Encoder-Decoder

Transform input to output

Attention: Encoder: bidirectional; Decoder: causal + cross-attention

Task: Sequence-to-sequence transformation

Used for: Translation, Summarization, Question Answering

Examples: T5, BART, FLAN-T5, mT5

Think of it as: A translator who reads the full source, then writes the target

Prompt Engineering — Complete Guide

You have access to a brilliant expert. But the quality of their answer depends entirely on how you ask the question. Ask vaguely, get a vague answer. Ask with context, structure, and examples — get a precise, expert-level answer. Prompt Engineering is the discipline of crafting inputs to LLMs to reliably produce high-quality, task-specific outputs.

❌ Bad Prompt

"Summarise this document."

Result: Generic 3-line summary with no structure, misses key points, wrong tone for audience.

✅ Good Prompt

"You are a financial analyst. Summarise this earnings report in 5 bullet points for a non-technical board audience. Focus on revenue, profit margin, and risks. Use plain language. Output format: bullet list."

📚 Prompting Techniques — From Basic to Advanced

1️⃣ Zero-Shot

No examples given

Prompt: "Classify this review as positive or negative: 'Great product, very fast delivery!'"

Model uses general knowledge. Works for simple, common tasks.

2️⃣ Few-Shot

Give 2–5 examples

Prompt: "Review: 'Excellent!' → Positive | Review: 'Terrible!' → Negative | Review: 'Decent quality' → ?"

Model learns the pattern from examples. Better for specific formats or niche tasks.

3️⃣ Chain of Thought (CoT)

Think step by step

Prompt: "If I have ₹10,000 and invest 40% in equities and 60% in FDs, how much is in FDs? Let's think step by step."

Forces reasoning. Dramatically improves math, logic, multi-step tasks.

4️⃣ Tree of Thought (ToT)

Explore multiple paths

Model generates multiple solution paths, evaluates each, backtracks if stuck. Best for complex planning and strategy problems. Like a decision tree in the model's reasoning.

5️⃣ ReAct

Reason + Act + Observe

Model interleaves reasoning with tool calls. Thought → Action (call API/search) → Observation → Next Thought. Powers AI agents that use the web, code execution, databases.

6️⃣ System Prompts

Define AI's persona and rules

Sent before conversation. Defines: role, constraints, output format, tone, what to refuse. Invisible to user. Example: "You are a SEBI-compliant financial advisor. Never recommend specific stocks."

🔧 Function Calling / Tool Use

Modern LLMs can decide which tool to call based on the user's request. You define available tools (database lookup, calculator, API call), and the model outputs a structured JSON specifying the tool and parameters — your code executes it and returns the result.

RAG — Complete Architecture & Implementation

An LLM knows everything it learned during training — but nothing after that. A GPT-4 model trained in April 2023 doesn't know about new products, recent regulations, or your internal company policies. When asked, it either says "I don't know" or worse — makes something up (hallucination).

RAG solves this by giving the model access to a real-time knowledge base. Before answering, the system retrieves the most relevant documents from your database and includes them in the prompt. The model reads the relevant context and answers based on actual facts — not training memory.

RAG PIPELINE — HOW IT WORKS: INDEXING PHASE (one-time, offline): ┌────────────────┐ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │ Your Documents │───▶│ Chunking │───▶│ Embedding │───▶│ Vector DB │ │ PDFs, Docs, │ │ Split into │ │ Model │ │ Store │ │ Policies, │ │ 500-word │ │ Text → │ │ Each chunk │ │ Knowledge Base │ │ chunks │ │ Vector │ │ + vector │ └────────────────┘ └──────────────┘ └──────────────┘ └──────────────┘ RETRIEVAL PHASE (real-time, per query): User Query │ ▼ Embed Query → Query Vector │ ▼ Search Vector DB (cosine similarity) → Top-K relevant chunks │ ▼ Build Augmented Prompt: "Context: [chunk1] [chunk2] [chunk3] Question: [user question] Answer based only on the context above." │ ▼ LLM generates answer grounded in YOUR documents │ ▼ Response (with source citations)

🔢 What Are Embeddings?

An embedding is a dense numerical vector that captures the meaning of text. "Heart attack" and "Myocardial Infarction" have different words but similar vectors — because they mean the same thing. Semantic search uses this — find documents by meaning, not just keyword match.

TEXT → EMBEDDING VECTOR (1536-dimensional): "Customer complaint about delayed delivery" → [0.234, -0.891, 0.456, 0.123, ..., -0.677] ← 1536 numbers "Order arrived late, very upset with service" → [0.241, -0.879, 0.461, 0.118, ..., -0.681] ← very similar! Cosine Similarity = 0.97 ← Nearly identical meaning "The GDP grew by 4.2% in Q3" → [-0.112, 0.334, -0.789, 0.901, ..., 0.234] ← very different Cosine Similarity vs complaint = 0.12 ← Unrelated topic

# ── STEP 2: RETRIEVE — Find relevant chunks for a query ──────────
def retrieve(query, top_k=2):
    """Find most semantically similar documents to the user's query"""
    # Embed the query
    query_vec = embed_texts([query])
    query_arr = np.array(query_vec[0])  # Convert to numpy array
    
    # Compute cosine similarity with all document vectors
    scores = []
    for i, doc_vec in enumerate(doc_vectors):
        doc_arr = np.array(doc_vec)
        # Cosine similarity = dot product of normalised vectors
        similarity = np.dot(query_arr, doc_arr) / (
            np.linalg.norm(query_arr) * np.linalg.norm(doc_arr)
        )
        scores.append((similarity, i))
    
    # Sort by similarity (highest first) and return top_k
    scores.sort(reverse=True)
    return [documents[i] for _, i in scores[:top_k]]

# ── STEP 3: GENERATE — Augmented prompt with retrieved context ────
def rag_answer(question):
    """Full RAG pipeline: retrieve relevant docs → generate grounded answer"""
    # Retrieve relevant documents
    relevant_docs = retrieve(question, top_k=2)
    context = "\n".join([f"[{i+1}] {doc}" for i, doc in enumerate(relevant_docs)])
    
    # Build augmented prompt
    augmented_prompt = f"""Answer the question using ONLY the provided context.
If the answer is not in the context, say "I don't have that information."

Context:
{context}

Question: {question}"""
    
    # Generate answer with LLM
    response = client.chat(ChatDetails(
        compartment_id=COMPARTMENT,
        serving_mode=OnDemandServingMode(model_id="cohere.command-r-plus"),
        chat_request=CohereChatRequest(
            message=augmented_prompt,
            preamble="You are an accurate AI assistant. Only answer from the provided context.",
            temperature=0.0,  # Zero temperature = deterministic, factual answers
            max_tokens=300
        )
    ))
    return response.data.chat_response.text

What is Fine-Tuning? And When Should You Use It?

A pre-trained LLM is like a brilliant new employee who has read every book ever written — but hasn't seen a single document from your company. They speak general English but don't know your product codes, your compliance terminology, or how your reports are formatted. Fine-tuning is the onboarding process — you show them hundreds or thousands of examples of your domain's inputs and expected outputs until they learn your style, vocabulary, and conventions.

🔷 What

Continuing to train a pre-trained model on your domain-specific dataset, adapting its weights to specialise for your task — without training from scratch.

⚡ Why
  • Model doesn't know your terminology
  • Inconsistent output format
  • Prompt engineering isn't enough
  • RAG can't change behaviour/style
  • Need task-specific personality
✅ When to Use
  • You have 500–100,000 labelled examples
  • Domain jargon not in base model
  • Consistent output format critical
  • Low-latency inference needed
  • Data privacy (no RAG retrieval)
⚠️ Common Mistake: Jumping straight to fine-tuning before trying prompt engineering or RAG. Fine-tuning is expensive. Always try simpler approaches first. Fine-tune only when prompt engineering and RAG provably fall short.

LoRA — Low-Rank Adaptation (The Industry Standard)

LoRA's insight: when you fine-tune a neural network for a specific task, the change in weights tends to lie in a low-dimensional subspace. You don't need to update 7 billion parameters — you only need to update a tiny low-rank approximation of those changes.

STANDARD FINE-TUNING: W_new = W_original + ΔW Problem: ΔW is the same shape as W (7B × 7B for large layers!) — expensive! LoRA: Instead of updating W directly, learn ΔW = A × B Where: A = (d × r) matrix ← small! e.g. 4096 × 16 B = (r × d) matrix ← small! e.g. 16 × 4096 r = rank (hyperparameter, usually 4–64) DURING TRAINING: Only A and B are updated (W is frozen) DURING INFERENCE: W_effective = W + (α/r) × A × B (Merged — zero extra cost at inference time!) RESULT FOR LLaMA-7B: Standard fine-tuning: update 7,000,000,000 parameters LoRA (r=16): update 4,194,304 parameters ← 0.06%! Memory saved: from 80GB GPU to 12GB GPU — revolutionary!
r (Rank)

Controls capacity

Higher r = more parameters trained = more adaptation capacity = more memory.
Common values: 4, 8, 16, 32, 64
Default: 16 for most tasks

lora_alpha

Scaling factor

Controls magnitude of LoRA update: scale = alpha/r.
Rule of thumb: alpha = 2×r
Effect: Higher alpha = LoRA weights matter more relative to base model

target_modules

Which layers to adapt

Typically: q_proj, v_proj (attention queries and values). Adding k_proj, o_proj, and FFN layers increases capacity but cost.
Typical: q_proj + v_proj

GRPO — Group Relative Policy Optimization

After supervised fine-tuning (SFT), models know your domain — but they may still produce poorly-reasoned, low-quality outputs. To teach a model to reason better, traditional RLHF (Reinforcement Learning from Human Feedback) uses a separate reward model trained on human preference pairs — expensive, complex, and slow.

GRPO (DeepSeek, 2024) eliminates the reward model entirely. It generates a group of responses for each prompt, scores them with an automatic rule-based function (correctness, format, length), and trains the model to prefer responses that score higher relative to the group average.

RLHF vs GRPO: RLHF: Prompt → LLM generates response → Human rates quality → Reward model learns preferences → PPO optimises LLM against reward model Problem: Need 2 LLMs running + expensive human labelling GRPO: Prompt → LLM generates GROUP of 8 responses → Auto-score each (rule-based) → Compute relative advantage: A_i = (score_i - mean_score) / std_score → Train model to increase probability of high-advantage responses WHY IT WORKS: - No reward model needed (saves 50% compute) - No human labelling needed (saves money) - Verifiable rewards: math = checkable, code = runnable, format = parseable - Used by DeepSeek-R1 to achieve o1-level reasoning at open-source cost
When to use GRPO
  • Teaching step-by-step reasoning
  • Math and logic tasks
  • Code generation (auto-test)
  • Structured output compliance
Reward Functions
  • Correctness: does answer match?
  • Format: valid JSON? proper structure?
  • Length: within specified limits?
  • Code: does it compile/run?

Quantization — Reducing Numerical Precision

Every weight in a neural network is stored as a floating-point number. By default: FP32 = 32 bits per parameter. A 7B model = 7,000,000,000 × 4 bytes = 28GB. That requires a GPU with 32GB+ VRAM — expensive and inflexible.

Quantization compresses these 32-bit numbers into 8-bit or 4-bit representations. 7B model in INT4 = 7B × 0.5 bytes = 3.5GB. The same model now runs on a laptop, a phone, or a cheap cloud VM — with only 1–3% accuracy loss.

PRECISION FORMATS — What the numbers look like: FP32 (32-bit): [sign][8-bit exponent][23-bit mantissa] Range: 1.4×10⁻⁴⁵ to 3.4×10³⁸ ← Extremely precise Memory: 4 bytes per weight FP16 (16-bit): [sign][5-bit exponent][10-bit mantissa] Range: 6×10⁻⁸ to 65,504 ← Less precise, faster Memory: 2 bytes per weight BF16 (16-bit "brain float"): [sign][8-bit exponent][7-bit mantissa] Range: Same as FP32, less mantissa precision — preferred for training! Memory: 2 bytes per weight INT8 (8-bit integer): Range: -128 to 127 ← Quantized representation Memory: 1 byte per weight INT4 / NF4 (4-bit): Range: -8 to 7 (16 values) ← Most aggressive compression Memory: 0.5 bytes per weight ┌────────────────────────────────────────────────────────┐ │ 7B Model Size by Precision: │ │ FP32 → 28 GB (training) │ │ FP16 → 14 GB (standard deployment) │ │ INT8 → 7 GB (quantized deployment) │ │ INT4 → 3.5 GB (edge/mobile deployment) │ └────────────────────────────────────────────────────────┘

🔬 Three Quantization Methods

1️⃣ Dynamic Quantization

Convert at inference time

Weights remain FP32, converted to INT8 on-the-fly during inference. Zero retraining needed. Good for quick deployment.

OCI Use: Quick deploy existing models on smaller VMs

2️⃣ Static Quantization

Pre-calibrate with data

Run representative data through model to compute activation ranges, then quantize. Better accuracy than dynamic. Requires calibration dataset.

OCI Use: Production inference with calibrated accuracy

3️⃣ QAT (Quantization-Aware Training)

Train with fake quantization

Simulate INT8 operations during training so model learns to work within quantized constraints. Best accuracy — model adapts its weights to quantization.

OCI Use: Mobile/edge deployment requiring best quality

Pruning — Removing What Doesn't Matter

Neural networks are massively over-parameterised — researchers have shown that 30–70% of weights can be set to zero with minimal accuracy loss. Like a garden: pruning dead branches doesn't kill the tree, it focuses its energy.

PRUNING TYPES: UNSTRUCTURED PRUNING (Weight-level): Original: [0.82, -0.01, 0.67, 0.003, -0.91, 0.002, 0.55] Pruned: [0.82, 0, 0.67, 0, -0.91, 0, 0.55] → Near-zero weights set to 0. Creates sparse model. Problem: Modern GPUs don't accelerate sparse operations well → limited speedup STRUCTURED PRUNING (Neuron/Head-level): Remove entire attention heads or neurons with lowest importance score. Remaining structure = dense (smaller) network → real hardware speedup! LAYER PRUNING: 7B model (32 layers) → Remove 8 lowest-importance layers → ~5.5B effective model Aggressive: needs fine-tuning after pruning to recover accuracy

Knowledge Distillation — The Teacher Trains the Student

A junior doctor learns more from shadowing an experienced surgeon than from reading textbooks alone. The surgeon doesn't just say "do this" — they explain their reasoning, show the alternatives they considered, and share their confidence levels. Knowledge Distillation applies this to AI: a large, accurate teacher model trains a small, fast student model by sharing not just correct answers but its probability distributions over all answers.

STANDARD TRAINING vs KNOWLEDGE DISTILLATION: Standard Training: Input → Student → Output: "cat" (hard label) Loss = CrossEntropy(prediction, "cat") ← learns only the right answer Knowledge Distillation: Input → Teacher → Soft labels: cat=85%, dog=10%, tiger=5% ← rich signal! Input → Student → Soft prediction: cat=72%, dog=18%, tiger=10% Loss = α × KL_Divergence(student_soft, teacher_soft) ← match teacher's uncertainty + (1-α) × CrossEntropy(student_pred, hard_label) ← also learn ground truth WHY SOFT LABELS ARE BETTER: Hard label "cat" gives 1 bit of information. Teacher's "cat=85%, dog=10%, tiger=5%" reveals: - This image looks somewhat like a dog too (maybe confused photo) - Tiger and cat share visual features (stripes? eyes?) Student learns the STRUCTURE of similarity, not just correct class!
Response Distillation

Match output distributions

Student matches teacher's final softmax output. Simplest form. Used by DistilBERT (40% smaller, 97% of BERT accuracy).

Feature Distillation

Match intermediate layers

Student also matches teacher's hidden state representations at each layer. More complex but more powerful — student "sees" teacher's thinking process.

Relation Distillation

Match sample relationships

Student learns how samples relate to each other in the teacher's embedding space — not just individual outputs but the geometry of understanding.

# ── KNOWLEDGE DISTILLATION: BERT-large → DistilBERT ─────────────
# OCI Use Case: Bajaj Finserv — Credit Risk Classifier on Android Tablets
import torch
import torch.nn as nn
import torch.nn.functional as F
from transformers import AutoModelForSequenceClassification

# ── Load Teacher (large, accurate, stays FROZEN during distillation) ──
teacher = AutoModelForSequenceClassification.from_pretrained(
    "./teacher_bert_large_credit",  # Fine-tuned BERT-large (1.3GB)
    num_labels=3  # Low Risk / Medium Risk / High Risk
)
teacher.eval()  # Teacher is inference-only — no gradient computation
for param in teacher.parameters():
    param.requires_grad = False  # Freeze teacher completely

# ── Load Student (small, fast, will be trained) ──────────────────
student = AutoModelForSequenceClassification.from_pretrained(
    "distilbert-base-uncased",  # DistilBERT: 66M params (vs BERT-large: 340M)
    num_labels=3
)

# ── Distillation Loss Function ────────────────────────────────────
def distillation_loss(student_logits, teacher_logits, true_labels, T=4.0, alpha=0.7):
    """
    Combined loss = alpha × KD_Loss + (1 - alpha) × CE_Loss
    
    T (temperature): Higher T → softer distributions → more signal transferred
    alpha: Weight on teacher knowledge (0.7 = 70% teacher, 30% hard labels)
    """
    # Soften both distributions by temperature T
    teacher_probs = F.softmax(teacher_logits / T, dim=-1)
    student_log_probs = F.log_softmax(student_logits / T, dim=-1)
    
    # KL Divergence: how different is student from teacher?
    kd_loss = nn.KLDivLoss(reduction="batchmean")(student_log_probs, teacher_probs)
    kd_loss *= (T ** 2)  # Scale back up (temperature divided gradients)
    
    # Standard cross-entropy against true labels
    ce_loss = nn.CrossEntropyLoss()(student_logits, true_labels)
    
    return alpha * kd_loss + (1 - alpha) * ce_loss

# ── Training Loop (simplified) ────────────────────────────────────
optimizer = torch.optim.AdamW(student.parameters(), lr=5e-5)

for batch in train_dataloader:
    inputs, labels = batch
    
    # Teacher prediction — no gradient needed
    with torch.no_grad():
        teacher_outputs = teacher(**inputs)
    
    # Student prediction — gradient flows through here
    student_outputs = student(**inputs)
    
    # Compute combined distillation loss
    loss = distillation_loss(student_outputs.logits, teacher_outputs.logits, labels)
    
    loss.backward()
    optimizer.step()
    optimizer.zero_grad()

Evaluating LLMs — Metrics, Methods & Hallucination

An LLM that confidently gives wrong answers is worse than one that says "I don't know." Before deploying any LLM in production — especially in finance, healthcare, or legal — you must rigorously evaluate its accuracy, reliability, and safety.

📊 BLEU / ROUGE

Text overlap metrics

Compare generated text to reference. BLEU for translation. ROUGE for summarisation. Limitation: penalise valid paraphrases.

🎯 Perplexity

Language model confidence

How surprised is the model by the test text? Lower perplexity = model understands this domain well. Used during pre-training evaluation.

🏆 RAGAS

RAG evaluation framework

Evaluates: Faithfulness (stays in context?), Answer Relevancy, Context Recall, Context Precision. Gold standard for RAG systems.

🤖 LLM-as-Judge

Model evaluates model

Use GPT-4 or Claude to rate another model's outputs on accuracy, helpfulness, safety. More nuanced than automated metrics.

⚠️ Hallucination — The #1 LLM Risk in Enterprise

Hallucination: An LLM confidently generates false information that sounds plausible. In a customer-facing banking chatbot, a hallucinated interest rate can cause regulatory violations. In a medical assistant, a hallucinated drug dosage can be life-threatening.

TYPES OF HALLUCINATION: 1. FACTUAL HALLUCINATION: User: "What is the current repo rate set by RBI?" Model: "The RBI repo rate is 5.75%" ← WRONG (confidently stated fiction) 2. SOURCE HALLUCINATION: Model: "According to RBI circular DBOD.No.BP.BC.2024/21.04.048/2024-25..." ← Circular reference may not exist! 3. CONTEXT HALLUCINATION (in RAG): Retrieved doc says: "Maximum loan tenure: 20 years" Model answers: "Maximum loan tenure: 25 years" ← Ignored/misread context! HOW TO MEASURE IT: Hallucination Rate = (Wrong factual claims) / (Total factual claims) × 100% HOW TO REDUCE IT: → RAG (ground in documents) — most effective → Temperature = 0.0 (deterministic, less creative, less hallucinatory) → System prompts: "Only answer from provided context" → Fine-tuning on domain to reduce uncertainty → Post-generation fact checking pipeline → Guardrails (filter outputs before user sees them)

💾 Memory Optimization — VRAM Planning

GPU VRAM REQUIREMENTS — Rule of Thumb: Required VRAM = model_size_GB × precision_multiplier × 1.25 (overhead) Where precision_multiplier: FP32 → 4 bytes/param | FP16 → 2 | INT8 → 1 | INT4 → 0.5 Examples: 7B FP16: 7B × 2 bytes × 1.25 = ~17.5 GB → need A100 40GB 7B INT4: 7B × 0.5 bytes × 1.25 = ~4.4 GB → need A10 24GB ✓ 70B INT4: 70B × 0.5 × 1.25 = ~43.75 GB → need A100 80GB KV-CACHE (add for inference batching): KV_cache_GB = 2 × layers × heads × head_dim × batch_size × seq_len × precision For 7B model, batch=8, seq=2048, FP16: ~2GB extra VRAM needed TOTAL for production serving: Model weights + KV cache + Optimizer states (for training) + Activations

🤖 Agentic AI

🖼️ Multimodal LLMs

Text + Image + Audio + Video

Single model processes text, images, tables, charts, PDFs, audio, video. GPT-4V, Claude 3 vision, Gemini 1.5 Pro. Enterprise: analyse invoices, contracts, and X-rays without separate models.

🧮 Reasoning Models

"Thinking" before answering

o1, o3, DeepSeek-R1 — models that generate internal chain-of-thought before answering. 2× to 10× better on math, coding, scientific reasoning. Cost: 5–10× more tokens consumed per query.

⚡ Efficient Models

Doing more with less

Phi-3, Gemma-2, Qwen-2.5 proving SLMs approach LLM quality on specific tasks. MoE (Mixture of Experts): activate only 2 of 8 expert networks per token — Mixtral 8×7B inference cost = 12B, not 56B.

📋 Regulation

AI governance is arriving

EU AI Act (2024): risk-based regulation. India DPDP Act. SEBI AI guidelines for BFSI. Enterprise AI teams need audit trails, explainability, and human oversight frameworks built in — not added later.

🔬 Test-Time Compute

Spend more tokens = smarter answers

Instead of bigger models, spend more compute at inference time. Generate multiple solution paths, vote on best, backtrack, self-verify. Same model, dramatically better accuracy on hard problems.
