DROP TABLE students PURGE;

-- Step 1B: Create the STUDENTS table
CREATE TABLE students (
  student_id    NUMBER         GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  student_name  VARCHAR2(100)  NOT NULL,
  email         VARCHAR2(150),
  department    VARCHAR2(50)   NOT NULL,
  created_date  DATE           DEFAULT SYSDATE NOT NULL
);
💡 Why GENERATED ALWAYS AS IDENTITY
The assignment specifically says: "use an Identity column (the modern, currently recommended approach) rather than a separate sequence-and-trigger."

GENERATED ALWAYS AS IDENTITY means Oracle automatically assigns the next unique number to STUDENT_ID every time you insert a row. You never supply this value yourself — Oracle handles it. This is the modern Oracle 12c+ approach.
Step 1C — Populate with 5 Records Using a PL/SQL Block
The assignment says: use a PL/SQL anonymous block — not five separate INSERT statements.

Step 5 · Validation Rules
5
Add Two Validation Rules
Validations
Key Requirement — Read Carefully
Rule A — Client-side: Error must appear in the browser instantly when submit is attempted — NO round trip to the server.

Rule B — Server-side: Error must only appear after the form is submitted to the server — cannot fire before that.

These are two completely different APEX mechanisms.
Rule A — Client-Side: Student Name Cannot Be Empty
Mechanism to use: Item → Value Required = Yes (HTML5 required attribute)

  1. In Page Designer, click on P[N]_STUDENT_NAME item.
  2. Right Panel → Validation section → Value Required: Yes.
  3. This adds the HTML5 required attribute to the input field.
  4. Save (Ctrl+S).

Why this is client-side: The browser itself checks the required attribute before the form is even submitted. The page never sends a request to the server — the browser blocks the submit and shows an error instantly.
💡 Alternative Client-Side Method
Another valid approach: Add a Dynamic Action on the Save button click event that uses JavaScript to check if the field is empty and show an alert — before page submit. Either approach satisfies Rule A as long as it prevents the server round trip.

Screenshot 3: Page Designer showing Rule A configured on P[N]_STUDENT_NAME — show the Value Required = Yes setting in the Right Panel.
Rule B — Server-Side: Letters and Spaces Only
Mechanism to use: APEX Validation → PL/SQL Expression

  1. In Page Designer → Left Panel → Processing tab → right-click Validating → Create Validation.
  2. Right Panel settings:
      • Name: Student Name Letters Only
      • Type: Function Body Returning Boolean (or PL/SQL Expression)
      • Item: P[N]_STUDENT_NAME
  3. PL/SQL Expression (use this): 
    REGEXP_LIKE(:PN_STUDENT_NAME, '^[A-Za-z ]+$')
    Replace N with your actual page number.
  4. Error Message: Student Name must contain only letters and spaces. — exact text required.
  5. Always Execute: No — only fire when student name is not null (Rule A handles the empty case).
  6. Save (Ctrl+S).

Why this is server-side: APEX Validations run during the Processing phase — only after the form has been submitted and the request has reached the server. The browser has no knowledge of this rule.
What REGEXP_LIKE Does Here
^[A-Za-z ]+$ means:

^ — Start of string
[A-Za-z ] — Only uppercase letters, lowercase letters, or a space
+ — One or more characters
$ — End of string

So Arjun Kumar → passes (only letters and space)
So John123 → fails (contains digits) → error shown

Step 6 · Dynamic Action — CSE Reminder Message
6
Show CSE Elective Reminder Instantly
Dynamic Action
What to Build
When Department = CSE → show message: "CSE students must register their elective by Friday."
When Department = anything else → hide the message.
Must happen instantly — no page reload.
6A — Add a Display Item for the Message
  1. In Page Designer, add a new item below the Department field.
  2. Type: Display Only (or Static Content / HTML region).
  3. Name: P[N]_CSE_MESSAGE.
  4. Default Value: CSE students must register their elective by Friday.
  5. Under Layout → set it as Hidden by default (so it starts invisible).
  6. Save (Ctrl+S).
6B — Create the Dynamic Action
  1. In Page Designer, right-click P[N]_DEPARTMENT item → Create Dynamic Action.
  2. Name: Show CSE Message.
  3. WHEN settings:
      • Event: Change
      • Selection Type: Item
      • Item: P[N]_DEPARTMENT
  4. Condition settings:
      • Condition Type: Equal to
      • Value: CSE
  5. True Action:
      • Action: Show
      • Affected Elements → Item: P[N]_CSE_MESSAGE
  6. Click + Add False Action:
      • Action: Hide
      • Affected Elements → Item: P[N]_CSE_MESSAGE
  7. Save (Ctrl+S).
💡 Also Hide on Page Load
Add a second Dynamic Action:
WHEN: Page Load → Action: Hide → Item: P[N]_CSE_MESSAGE

This ensures the message is hidden when the form first opens — before the user picks any department.
